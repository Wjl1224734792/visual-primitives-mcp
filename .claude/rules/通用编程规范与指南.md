# 通用编程规范与指南

## 1. 开发环境配置

- **语言**：文档、注释、沟通强制使用**中文**。
- **终端**：Windows 推荐 PowerShell；生产环境 Linux。

## 2. 代码规范

### 2.1 注释 (JSDoc/TSDoc)

- 只写关键逻辑与边界条件，必须包含 `@param`、`@returns`、`@throws`。

### 2.2 嵌套层级

- 最多 4 层，禁止超过。用卫语句、函数提取优化。

### 2.3 数组操作

- **禁止**：`push`, `pop`, `shift`, `splice`, `sort`, `reverse`（空数组初始化除外）。
- **推荐**：`map`, `filter`, `reduce`, `toSpliced`, `toSorted`, `with`, 扩展运算符。

### 2.4 模块化

- 推荐命名导出，路径使用别名（如 `@/`），禁止循环依赖。

### 2.5 设计原则

- **核心原则**：SOLID、DRY、KISS。
- **条件优化**：3 个以上分支用 Map/对象映射；强制 `===`；使用 `??` 和 `?.`。
- **箭头函数**：禁止在对象/类方法中使用。
- **Promise**：善用 `Promise.all`，避免无意义 `return await`。

### 2.6 DDD（领域驱动设计）

- **适用**：仅复杂业务逻辑（状态流转、多聚合交互）。
- **战术**：聚合根（业务边界）、值对象（不可变）、领域服务（跨聚合）、领域事件（解耦）。
- **禁止**：数据访问逻辑放入实体；应用层编写核心业务规则。

### 2.7 TDD（测试驱动开发）

- **流程**：红（失败测试）→ 绿（最快通过）→ 重构。
- **要求**：新增核心业务逻辑必须先写单元测试；每个测试验证一个行为。
- **例外**：UI 快照、集成测试不强制，但关键流程仍需 TDD。

### 2.8 数据库外键

- **禁止物理外键**，ORM 逻辑关联但禁用约束（如 `createForeignKeyConstraints: false`）。

### 2.9 Tailwind CSS

- **禁止 `@apply`**，只能使用内联类名（如 `px-4 py-2`）。

## 3. 质量保证

### 3.1 例外

- 算法递归、性能关键代码需注释并团队评审。

### 3.2 强制检查项（Commit 前）

1. 嵌套 ≤4 层
2. 无循环依赖
3. 多分支用 Map，判空用 `?.`，全等 `===`
4. Promise 正确处理错误
5. 无物理外键
6. 无 `@apply`
7. 运行 `lint`、`typecheck`、`build`（若有）、`test`，全部通过
8. 核心业务规则在聚合根/领域服务，不泄露到应用层
9. 新增核心逻辑附带单元测试

## 4. 沟通风格

- 友好表达（适当 Emoji），❌错误 vs ✅正确对比，结构化响应（问题→方案→实践→注意）。
