# 增量实现

## 概述

增量实现的核心是：**每次只交付一个可独立验证的最小价值单元**。不是把整个功能写完才测试——而是每 10-20 行代码就验证一次。

**核心原则：** 小增量 = 小风险。大爆炸式提交 = 大风险。每步都可以验证 → 每步都可以回退 → 每一步都是安全的。

**必需背景：** 你必须理解 `behavioral-guidelines`（准则 2：简单优先）和 `test-driven-development`（增量实现的黄金搭档）。

## 何时使用

**适用场景：**

- 实现任务超过 50 行代码
- 修改涉及多个文件或模块
- 在陌生代码库中工作
- 涉及高风险变更（支付、认证、数据迁移）
- 多人并行开发，需要频繁集成

**不适用场景：**

- 单文件单函数的小改动（<30 行）
- 纯配置变更
- 格式化和 lint 修复
- 搜索替换式的重命名

## 增量拆分策略

### 策略 1：由内向外

先实现最底层的核心逻辑，再逐步向上构建接口和 UI。

```
第 1 步：数据模型 + 验证逻辑  →  测试
第 2 步：数据访问层           →  测试
第 3 步：业务服务层           →  测试
第 4 步：API 路由 + 请求处理   →  测试
第 5 步：前端组件 + 集成      →  测试
```

### 策略 2：由外向内（推荐用于 API 开发）

先定义契约（接口/类型/schema），再从两端向中间实现。

```
第 1 步：定义 API 契约（输入/输出类型）    →  lint 检查
第 2 步：实现 API 路由 + 模拟响应           →  手动验证
第 3 步：实现数据层                         →  测试
第 4 步：实现业务逻辑，替换模拟响应          →  测试
```

### 策略 3：垂直切片

每步交付一个完整的端到端子功能（见 Jarvis 的垂直切片原则）。

```
第 1 步：用户注册（schema + API + 基本 UI）  →  独立验证
第 2 步：用户登录（auth + API + 基本 UI）     →  独立验证
第 3 步：密码重置（schema + API + 基本 UI）   →  独立验证
```

### 策略 4：特性开关驱动

每个增量用特性开关包裹，未完成的功能不暴露给用户。

```
第 1 步：[FEATURE_FLAG] 基础数据模型        →  仅开发环境可见
第 2 步：[FEATURE_FLAG] API 端点             →  仅内部测试可见
第 3 步：[FEATURE_FLAG] 完整 UI + 集成       →  灰度放开
```

## 每步增量的完成标准

每次增量提交前必须满足：

1. **可编译/可运行**：代码不处于"写了一半"的中间状态
2. **相关测试通过**：如果有 TDD 策略，该步的测试已通过
3. **lint/typecheck 通过**：不引入新的 lint 或类型错误
4. **独立可验证**：该增量本身可以展示或测试（即使整体功能未完成）
5. **变更可追溯**：commit message 关联到 `TASK-XXX` 或 `REQ-XXX`

---

## 不可减少的步骤

以下步骤在任何增量中都不应跳过：

1. **先读后写**：每次修改文件前，先读一遍现有内容（`source-driven-development`）
2. **验证后提交**：每步改完立即验证（lint + typecheck + 相关测试），不要积攒到最后
3. **小步提交**：每 1-5 个逻辑上关联的文件修改为一个 commit
4. **commit message 标记**：标注对应的 TASK-XXX，方便追溯

## 增量反模式

| 反模式           | 说明                                   | 后果                           |
| ---------------- | -------------------------------------- | ------------------------------ |
| **大爆炸提交**   | 把所有改动攒成一个 commit              | Review 困难，回滚困难，冲突多  |
| **半成品提交**   | commit 的代码无法编译/跑不起来         | 破坏 bisect，阻塞其他开发者    |
| **跨关注点提交** | 一个 commit 同时包含功能、重构、格式化 | 难以 review，难以 cherry-pick  |
| **无验证提交**   | commit 后不跑 lint/test                | 引入你不知道的回归             |
| **过早抽象**     | 第一步就创建抽象层、工具函数           | 抽象建立在猜测之上，大概率要改 |

## 常见借口

| 合理化借口                 | 现实                                                                |
| -------------------------- | ------------------------------------------------------------------- |
| "改动都是关联的，拆不开"   | 关联的改动正是最需要分步验证的。拆不开 = 耦合太高，更需要小步重构。 |
| "功能很简单，一次写完就行" | 简单功能也会引入意外 bug。小步提交让你能精确知道哪一步引入了问题。  |
| "频繁提交太麻烦了"         | 每次提交的成本远低于排查一个大 commit 出问题的成本。                |
| "分了步不好 review"        | 恰恰相反——分步 review 可以看每一步做了什么、为什么这么做。          |
| "20 行代码不需要拆分"      | 好。20 行不需要。但 200 行需要。判断标准：超过 50 行就考虑拆分。    |

## 红线——停下来，按流程走

如果你发现自己在想：

- "这些改动都差不多，攒一起提交吧"
- "还没完全调通，先 commit 保存一下"
- "重构+功能写一起，一次提交省事"
- "改完了，一起跑测试就行"
- "先把所有文件的框架搭好，再逐个填充"

**以上这些都意味着：你没有在做增量实现。回到拆分策略，把任务分解为更小的可验证步骤。**

## 验证清单

每次增量提交前确认：

- [ ] 变更范围清晰（改动不超过 5 个文件或 200 行）
- [ ] 代码可编译/可运行
- [ ] lint/typecheck 通过
- [ ] 相关测试通过（改了什么跑什么，不是全量跑）
- [ ] 该增量可独立验证（有明确的验收点）
- [ ] commit message 关联到 TASK-XXX
- [ ] 没有夹带无关重构或格式化
- [ ] 共享区域没有被意外修改
