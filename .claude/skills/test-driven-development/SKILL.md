# 测试驱动开发（TDD）

## 概述

TDD 不是"先写测试"，而是**用测试驱动设计**。三个步骤形成一个不可倒置的闭环：Red（写失败测试）→ Green（写最小实现）→ Refactor（重整代码，测试仍绿）。

**核心原则：** 没有测试就没有实现。测试是需求的可执行版本——它先于代码存在，定义"什么算完成"。

**必需背景：** 你必须理解 `behavioral-guidelines`（准则 4：目标驱动执行）。

## 何时使用

**适用场景：**

- test_strategy 被标注为 `tdd` 的任务
- 新功能的实现（测试先于代码）
- Bug 修复（先写重现 bug 的测试，再修）
- 重构（已有测试保护，重构后仍绿）
- API 端点实现（先定义契约测试）
- 业务规则实现（先写规则验证测试）

**不适用场景：**

- test_strategy 为 `manual_only` 的任务
- 纯 UI 样式调整（用视觉回归测试替代）
- 探索性原型（prototype/spike，用完即弃）
- 配置文件、文档、迁移脚本等非逻辑变更

## TDD 三阶段流程

```
RED ────────────→ GREEN ────────────→ REFACTOR
^                                        │
└────────────────────────────────────────┘
           (循环，直到功能完成)
```

### Red：写一个失败的测试

**目标：** 定义一个尚未满足的行为。

1. **先明确要测什么**
   - 一次只测一个行为（一个断言或一组紧密相关的断言）
   - 测试命名格式：`<被测方法>_<场景>_<预期结果>`
   - 例如：`createUser_WithDuplicateEmail_ThrowsConflictError`

2. **写出测试代码**
   - 使用仓库现有的测试框架和命名规范
   - Mock 外部依赖（数据库、HTTP、文件系统），不 mock 被测单元本身
   - 测试必须能独立运行，不依赖其他测试的执行顺序

3. **运行测试，确认失败**
   - 运行测试命令，保留失败输出
   - 确认失败原因是"功能尚未实现"而非"测试写错了"
   - 如果测试直接通过 → 说明这个行为已经存在，重新审视测试是否冗余

### Green：写最小实现让测试通过

**目标：** 用最少代码通过测试。

1. **只写刚好能让测试通过的代码**
   - 不要为"将来可能需要"写代码
   - 不要为"其他测试可能会测"写代码
   - 不要在这一步做重构或优化
   - 允许硬编码、允许重复、允许不优雅——先让测试绿

2. **运行测试，确认通过**
   - 只运行相关的测试，确认变绿
   - 再运行全量测试，确认没有破坏其他测试
   - 保留测试通过的输出作为验证证据

### Refactor：重整代码

**目标：** 消除 Green 步骤的临时粗糙，保持测试全绿。

1. **在测试保护下改进代码**
   - 消除 Green 步骤引入的重复
   - 改善命名（变量、函数、类型）
   - 提取公共逻辑
   - 优化算法（如有性能问题）

2. **每改一步就跑一次测试**
   - 保持测试始终绿色
   - 如果测试变红 → 立即回退上一步改动
   - 不要在测试红的状态下继续改

3. **Refactor 的边界**
   - 只重整本次改动涉及的范围
   - 不要顺手重构不相干的代码
   - 如果发现不相干代码需要重构 → 记录为独立任务，不在本次做

## TDD 常见模式

### 模式 1：三角测量（Triangulation）

先写一个最简单的测试→用硬编码通过→写第二个不同参数的测试→被迫去硬编码→写第三个→抽象出通用逻辑。

### 模式 2：假实现→真实现（Fake It → Make It）

先用返回常量的假实现通过测试，再逐步替换为真实逻辑。

### 模式 3：明显实现（Obvious Implementation）

如果实现方案一目了然（如简单的 getter/setter），可直接写出真实现，跳过假实现。

## 测试反模式——避免以下行为

| 反模式              | 说明                             | 正确做法                           |
| ------------------- | -------------------------------- | ---------------------------------- |
| **测试依赖顺序**    | 测试 A 创建的数据被测试 B 使用   | 每个测试独立准备数据，独立清理     |
| **测试私有方法**    | 直接测试 private 方法            | 通过公共接口间接覆盖               |
| **过度 mock**       | 连被测单元的内部方法也 mock      | 只 mock 外部边界（DB、HTTP、文件） |
| **断言太多**        | 一个测试里测了 5 个不相关的行为  | 每个测试只验证一个行为             |
| **测试间共享状态**  | 多个测试共享一个可变对象         | 每个测试创建自己的 fixture         |
| **不可靠测试**      | 依赖时间、随机数、网络           | 注入可控的时间/随机源，mock 网络   |
| **测试测的是 mock** | 验证 mock 被调用而非验证行为结果 | 断言输出/状态变化，不断言调用次数  |
| **不清理**          | 测试在 DB/文件系统留垃圾         | 每个测试后清理，或用事务回滚       |

## 测试分层策略

```
         ╱  E2E 测试 ╲         少（慢，脆，验证核心用户旅程）
        ╱  集成测试   ╲        中（验证组件间协作）
       ╱   单元测试    ╲       多（快，稳，验证业务逻辑）
```

- **单元测试（70%）**：测试单个函数/方法/类的行为
- **集成测试（20%）**：测试多个组件协作（API + DB、Service + Repository）
- **E2E 测试（10%）**：测试完整用户旅程（登录→操作→退出）

## 常见借口

| 合理化借口                     | 现实                                                                   |
| ------------------------------ | ---------------------------------------------------------------------- |
| "这个功能太简单了，不需要测试" | 简单功能的回归最容易被忽略。一行代码也能出错。                         |
| "先写实现，后面再补测试"       | "后面"永远不会来。测试先行的核心价值是设计反馈。                       |
| "测试跑得慢，浪费时间"         | 慢的测试是集成/E2E 测试。单元测试应该在毫秒级。分层测试，快的多写。    |
| "代码改了测试也得改，麻烦"     | 测试改得多说明测试测的是实现而非行为。行为不变测试不该变。             |
| "AI 生成的代码不用测"          | AI 代码比人写的更需要测试——AI 会有自信地犯错。                         |
| "我手动测过了，没问题"         | 手动测试不可重复、不可扩展。今天手动过了，明天改了相关代码没人会重测。 |

## 红线——停下来，按流程走

如果你发现自己在想：

- "这个测试感觉没必要写"
- "实现太简单了，直接写吧不用先写测试"
- "先写实现，测试一起补"
- "Red 步骤失败是因为测试写错了，但 Green 写对了就行"
- "Refactor 的时候顺便优化一下旁边的代码"
- "测试都通过了，不用再 Refactor 了"
- "这个 mock 太复杂了，直接测真实数据库吧"

**以上这些都意味着：回到 Red 步骤，重新开始。**

## 验证清单

每次 TDD 循环完成后确认：

- [ ] Red 阶段有失败测试的输出记录
- [ ] Green 阶段只写了刚好能通过测试的最小代码
- [ ] Green 阶段通过后运行了全量测试，无新增失败
- [ ] Refactor 阶段每步改动都保持了测试绿色
- [ ] Refactor 未涉及本次范围外的代码
- [ ] 测试命名清晰描述了被测行为
- [ ] 测试可独立运行，不依赖执行顺序
- [ ] 外部依赖已正确 mock
- [ ] 无测试反模式（依赖顺序、过度 mock、共享状态等）
- [ ] 新增代码的测试覆盖率满足项目标准
