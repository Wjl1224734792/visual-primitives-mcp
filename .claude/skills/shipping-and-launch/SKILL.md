# 发布与上线

## 概述

代码写完了、测试通过了、评审也过了——这只能说明代码在开发环境没问题。生产环境是另一回事。上线的核心不是"把代码推上去"，而是**以可控的风险把变更暴露给真实用户，并具备即时止损的能力**。

**核心原则：** 每一次上线都假设会出问题，提前准备好发现问题和撤销变更的手段。

**前置条件：** 本技能在 Gate D（评审通过）之后启动。如果 Gate D 尚未通过，退回 jarvis.md 流程补齐。

**开始时宣布：** "我正在使用 shipping-and-launch 技能准备上线上线。"

## 何时使用

**必须使用的情况：**

- 任何代码即将部署到生产环境
- 涉及数据库迁移、配置变更、第三方依赖更新的部署
- 新功能首次面向用户开放
- 涉及支付、认证、数据删除等高风险领域

**不需要使用的情况：**

- 纯文档更新（README、CHANGELOG）
- 仅影响开发环境的配置变更
- 已经验证过的热修复（第 N 次同路径部署，流程已固化）

## 流程

### 步骤 1：上线前检查清单

从头到尾跑一遍，任何一个不通过都不能继续。

#### 1.1 自动化检查

```bash
# 测试
<项目测试命令>  # 必须全绿

# Lint
<项目 lint 命令>  # 必须 0 warning

# 构建
<项目构建命令>  # 必须 exit 0
```

**如果任何一项失败：** 停止。回到实现修复。不要说"lint 警告不重要"。

#### 1.2 依赖审计

```bash
# Node.js
npm audit --production

# Python
pip-audit
# 或
safety check

# Go
govulncheck ./...
```

**关键依赖有已知漏洞 →** 评估影响，决定是否升级或接受风险（需记录）。

#### 1.3 安全扫描（如项目已配置）

```bash
# 常见工具
trivy fs .           # 容器/文件系统扫描
gitleaks detect      # 密钥泄露检查
semgrep --config=auto .  # 语义代码扫描
```

| 指标     | 上线前 | 上线后（观察期） | 差异 |
| -------- | ------ | ---------------- | ---- |
| P50 延迟 |        |                  |      |
| P99 延迟 |        |                  |      |
| 错误率   |        |                  |      |
| 吞吐量   |        |                  |      |
| 内存/CPU |        |                  |      |

### 步骤 2：功能开关规划

**不是所有功能都需要开关，但任何不确定的功能都需要。**

#### 2.1 判断是否需要功能开关

| 场景                             | 需要开关？                       |
| -------------------------------- | -------------------------------- |
| 新功能，用户可感知               | 强烈建议                         |
| 重构，行为不变                   | 可选（用金丝雀替代）             |
| Bug 修复                         | 通常不需要（除非修复方案有争议） |
| 基础设施变更（数据库/缓存/队列） | 强烈建议                         |
| 第三方 API 切换                  | 需要                             |

#### 2.2 功能开关生命周期

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│  OFF     │ → │ 1% 灰度  │ → │ 10% 灰度 │ → │ 50% 灰度 │ → │ 100%    │ → │ 清理    │
│ (默认关闭)│    │ (内部/种子)│    │ (小范围)  │    │ (半量)    │    │ (全量)   │    │ (移除开关)│
└──────────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘
     ↓               ↓               ↓               ↓               ↓               ↓
  开发完成      观察 1 小时     观察 4 小时     观察 24 小时     观察 1 周      确认稳定后
                无异常后推进     无异常后推进     无异常后推进     无异常后推进     移除开关代码
```

**每个阶段推进前必须确认：**

- 错误率未上升
- P99 延迟未恶化
- 无新增告警
- 用户反馈正常

**推进节奏准则：**

- 工作日白天推进，不在周五下午和节假日前上线
- 每个阶段留足观察时间，不跳跃
- 任何阶段出现异常 → 立即回滚开关至 0%，不试图"再观察一下"

#### 2.3 开关实现要点

```
# 好：开关有默认值、支持动态切换、有超时兜底
if featureFlag.isEnabled('new-checkout', userId, default = false):
    return newCheckout()
else:
    return legacyCheckout()

# 坏：硬编码、无兜底
if featureFlag.isEnabled('new-checkout'):  # 服务挂了直接炸
    return newCheckout()
```

#### 2.4 开关清理承诺

在功能开关全量稳定运行 N 天后（建议 7 天），**必须在下一个迭代中清理开关代码**。开关是债务，不是遗产。

在计划文档中写明清理时间和负责人：

```
开关清理承诺：
- 开关名称：new-checkout
- 全量时间：2026-05-01
- 清理截止：2026-05-08
- 负责人：@张三
```

### 步骤 3：选择上线策略

#### 3.1 策略选择矩阵

| 策略           | 适用场景                   | 回滚速度       | 成本           |
| -------------- | -------------------------- | -------------- | -------------- |
| **滚动更新**   | 常规无状态服务             | 分钟级         | 低             |
| **金丝雀部署** | 新版本风险不确定           | 秒级（切流量） | 中             |
| **蓝绿部署**   | 数据库变更、高风险上线     | 秒级（切流量） | 高（双倍资源） |
| **功能开关**   | 新功能灰度、A/B 测试       | 秒级（关开关） | 低             |
| **暗发布**     | 后端接口先行、前端延迟发布 | 分钟级         | 低             |

#### 3.2 金丝雀部署

```
1. 部署新版本到 1 个实例（金丝雀）
2. 将 1% 流量导向金丝雀
3. 观察 15-30 分钟
4. 逐步增加流量：1% → 5% → 25% → 50% → 100%
5. 每一阶段对比指标，异常则回退
```

#### 3.3 蓝绿部署

```
1. 准备绿色环境（新版本完整部署，但不接流量）
2. 在绿色环境跑冒烟测试
3. 切换负载均衡器 → 所有流量到绿色
4. 蓝色环境保留 24 小时（快速回滚用）
5. 确认稳定后回收蓝色
```

#### 3.4 选择依据

- 有数据库迁移 → 蓝绿（避免 Schema 不兼容导致回滚困难）
- 无状态服务 → 滚动更新足够
- 首次上线的大功能 → 金丝雀 + 功能开关组合
- 只有前端变更 → 功能开关

### 步骤 4：回滚预案

**没有回滚方案的上线就是赌博。**

#### 4.1 回滚判断矩阵

| 触发条件               | 响应                         | 时效要求   |
| ---------------------- | ---------------------------- | ---------- |
| 错误率超过基线 2 倍    | 自动回滚（如配置了自动告警） | 5 分钟内   |
| P99 延迟超过基线 3 倍  | 评估后决定                   | 15 分钟内  |
| 用户报告核心流程不可用 | 立即手动回滚                 | 发现即执行 |
| 数据库迁移失败         | 停止上线，恢复备份           | 立即       |
| 第三方 API 异常        | 降级方案 or 回滚             | 15 分钟内  |
| 内存/CPU 持续飙升      | 评估后决定                   | 30 分钟内  |

#### 4.2 回滚步骤文档（必须写入上线文档）

```
回滚方案：
1. 谁执行：@李四（主）/ @王五（备）
2. 如何回滚：
   - 功能开关：关闭开关 new-feature → 0%
   - 金丝雀：将流量切回旧版本
   - 蓝绿：将负载均衡切回蓝色环境
   - 数据库：执行迁移回滚脚本（如已准备）
   - 无上述手段：执行 CI/CD 回滚 pipeline
3. 回滚后验证：
   - 确认错误率恢复到上线前基线
   - 确认核心流程可用
   - 验证至少 3 个关键接口正常
4. 回滚审批：无需审批（回滚不应有阻力）
```

**关键原则：回滚不需要审批。** 出问题时一线执行者应有即时回滚权限。回滚后需要写复盘报告，但不需要事前报批。

### 步骤 5：监控与告警

#### 5.1 四项黄金指标

上线前必须确认以下四项在监控面板上可见：

| 指标       | 含义                           | 告警阈值建议   |
| ---------- | ------------------------------ | -------------- |
| **延迟**   | 请求耗时（P50 / P95 / P99）    | P99 > 基线 × 2 |
| **流量**   | 请求速率（QPS）                | 偏离基线 ± 50% |
| **错误率** | 5xx / 总请求                   | > 1%           |
| **饱和度** | CPU / 内存 / 连接池 / 队列深度 | > 80%          |

#### 5.2 上线专用监控面板

为本次上线创建临时 Dashboard，至少包含：

```
面板名称：launch-<功能名>-<日期>
指标：
  - QPS（按端点分组）
  - 错误率（按端点分组）
  - P50 / P95 / P99 延迟（按端点分组）
  - 功能开关状态
  - 健康检查状态
  - 关键业务指标（本次上线相关的）
```

#### 5.3 告警配置

```
告警规则：
- 错误率 > 1%，持续 2 分钟 → PagerDuty/钉钉/飞书
- P99 > 基线 × 2，持续 5 分钟 → 告警群
- 健康检查连续失败 3 次 → 立即通知
- 功能开关异常（服务不可达）→ 立即通知
```

#### 5.4 值班确认

- 上线时段必须有值班人员在线
- 值班人员知道本次上线的回滚方案
- 告警渠道已测试可送达

### 步骤 6：执行上线

按以下顺序执行，不允许跳跃：

```
1. 发出上线通告 → 通知相关团队（15 分钟前）
2. 确认检查清单全部通过
3. 执行部署（按选定的上线策略）
4. 打开监控面板，开始直播观察
5. 逐一验证关键功能可用
6. 如使用功能开关：先 0%，然后逐步灰度
7. 观察至少 30 分钟稳定后再离开
```

**上线通告模板：**

```
上线通告：
- 时间：2026-04-29 15:00
- 内容：新支付流程上线（REQ-012）
- 策略：金丝雀部署，1% → 10% → 50% → 100%
- 回滚方案：功能开关 payment-v2 切回 0%
- 监控面板：<Dashboard 链接>
- 值班人：@张三、@李四
- 预计影响：无（滚动上线，功能开关控制）
```

### 步骤 7：上线后观察

#### 7.1 观察时间表

| 时间段         | 活动                         |
| -------------- | ---------------------------- |
| 0-15 分钟      | 持续盯 Dashboard，确认无异常 |
| 15-30 分钟     | 验证核心功能，对比上线前基线 |
| 30 分钟-2 小时 | 每 15 分钟检查一次指标       |
| 2-4 小时       | 每 30 分钟检查一次           |
| 4-24 小时      | 每小时检查一次               |
| 24-72 小时     | 每日检查，确认无延迟异常     |

#### 7.2 上线后验证清单

- [ ] 错误率在基线的 1.5 倍以内
- [ ] P99 延迟未显著恶化
- [ ] 核心业务流程可正常走通（至少 3 个场景）
- [ ] 新增功能的用户反馈正常（如有反馈渠道）
- [ ] 数据库连接池、缓存命中率正常
- [ ] 第三方依赖调用正常
- [ ] 无内存泄漏迹象（观察内存曲线）

#### 7.3 异常处理

上线后 24 小时内发现异常：

1. 判断严重程度 → 决定回滚还是 hotfix
2. 如果回滚 → 执行回滚方案 → 通知团队 → 写复盘
3. 如果 hotfix → 走简化流程（仍需 Code Review + 测试）

## 常见合理化借口

| 合理化借口                       | 现实                                                   |
| -------------------------------- | ------------------------------------------------------ |
| "这就是个小改动，不需要上线检查" | 小改动也能炸。上线检查是肌肉记忆，养成习惯。           |
| "功能开关太麻烦，直接全量"       | 失去回滚能力。功能开关是保险，不是装饰。               |
| "我们上线后盯着就行"             | 人的注意力 30 分钟后下降。自动化监控是必须的。         |
| "上次没问题，这次也不会"         | 每次部署都是新的。环境变了，依赖变了，你变了。         |
| "回滚方案我知道，不用写"         | 真出问题时你脑子是乱的。写下来比记得更可靠。           |
| "监控面板太多，懒得建"           | 没有监控的上线等于蒙眼开车。出问题了都不知道。         |
| "金丝雀部署太慢，直接全量吧"     | 快 5 分钟上线 vs 回滚一下午。算算账。                  |
| "我们的系统很稳定，不会挂"       | 每个事故后的第一句话都是"我以为不会"。                 |
| "功能开关以后再清理"             | "以后" = "永远不会"。开关堆积是线上事故的定时炸弹。    |
| "不用发通告，没人会注意"         | 依赖方崩溃时第一个找的就是你。通告是免责声明。         |
| "这次只改配置文件"               | 配置错误是生产故障的第一大原因。配置变更也是变更。     |
| "部署完成了就算上线成功"         | 部署完成 ≠ 功能正常 ≠ 用户体验良好。上线成功需要验证。 |

## 与 jarvis.md 的衔接

本技能是 jarvis.md 主流程的后续扩展：

```
jarvis.md Gate D（评审通过）
     │
     ├── qa-review-expert 给出通过意见
     │
     ▼
shipping-and-launch（本技能）
     │
     ├── 上线前检查 → 功能开关 → 上线策略 → 回滚预案 → 监控
     │
     ▼
生产环境
```

**前置条件：** Gate D 必须已通过。如果 qa-review-expert 提出了阻塞性问题，必须先修复后重新评审，再进入上线流程。
