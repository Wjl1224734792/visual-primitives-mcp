# 通用编程规范

此技能适用于所有编写代码的实现类 agent（implementer、worker）。审查类 agent 按此规范检查合规性。

---

## 1. 注释规范

- 关键逻辑与边界条件必须编写文档注释，含 `@param`、`@returns`、`@throws` 等标记
- 注释语言统一使用中文（见 `behavioral-guidelines` 准则 5）

## 2. 嵌套层级

- 最多 4 层，禁止超过。用卫语句、函数提取等手段优化。

## 3. 集合/数组操作

- 优先不可变风格：`map`、`filter`、`reduce`、拼接、切片、展开运算符等返回新集合的操作
- 避免直接修改入参或原集合（`push`、`pop`、`remove`、`sort` 等副作用操作），除非有明确性能数据支撑且已评审

## 4. 模块化

- 推荐命名导出/导入（具名导出），路径使用别名（如 `@/`）
- 禁止循环依赖

## 5. 设计原则

- **核心原则**：SOLID、DRY、KISS
- **条件优化**：≥3 个分支时，优先使用映射表（Map/Dict/对象字面量）代替 if-else 或 switch
- **相等比较**：强制使用严格相等（`===` / 语言等价方式），避免隐式类型转换
- **空值处理**：使用 `??`、`?.`、Optional 类型等空安全操作，减少深层空检查
- **异步处理**：充分利用 `Promise.all`/`Future.sequence` 等并发组合，避免无意义的顺序等待；不必要时避免 `return await`

## 6. DDD（领域驱动设计）

- **适用**：仅复杂业务逻辑（状态流转、多聚合交互）
- **战术**：聚合根（业务边界）、值对象（不可变）、领域服务（跨聚合）、领域事件（解耦）
- **禁止**：数据访问逻辑放入实体；应用层编写核心业务规则

## 7. TDD（测试驱动开发）

- **流程**：红（失败测试） → 绿（最快通过） → 重构（`test-driven-development` 技能详述）
- **要求**：新增核心业务逻辑必须先写单元测试；每个测试验证一个行为

## 8. 数据库外键

- **禁止物理外键**，ORM 仅维护逻辑关联但禁用约束（如 `createForeignKeyConstraints: false`）

---

## 强制检查项（实现完成前必须执行）

调用 `verification-before-completion` 技能自查，额外确认：

1. 嵌套 ≤ 4 层
2. 无循环依赖
3. 多分支用映射，判空用空安全操作，严格相等
4. 异步/并发错误正确处理
5. 无物理外键（后端/数据层 agent）
6. Lint + Type-check + Build 全部通过（`code-quality-gate` 技能）
7. 核心业务规则在聚合根/领域服务，不泄露到应用层
8. 新增核心逻辑附带单元测试
