# 源码驱动开发

## 概述

"先读后写"不是慢，是快。凭记忆写的代码大概率与现状不兼容——类型不匹配、风格不一致、破坏了隐式契约。源码驱动开发强制你在动手之前，先充分理解目标代码的上下文。

**核心原则：** 代码不是你想的那样——它是它实际写的那样。读代码是唯一了解真相的方式。

**必需背景：** 你必须理解 `behavioral-guidelines`（准则 1：先思考再编码）和 `context-engineering`（在正确时间给正确信息）。

## 何时使用

**适用场景（几乎总是）：**

- 修改你不熟悉的代码文件
- 在超过 3 个文件的项目中修改
- 跨模块的实现任务
- 修改共享组件、工具函数、类型定义
- 对接已有的 API 或数据库 Schema
- 实现一个新功能但需要集成到现有架构中

**不适用场景：**

- 创建全新的独立文件，且不依赖现有代码
- 修改变量名之类的机械操作（但仍需 `grep` 确认影响范围）
- 配置文件的新增条目（但需要看现有条目格式）

## 阅读流程

### 第一步：读根契约

在修改任何代码之前：

1. **读取根 AGENTS.md**（如果有的话）
   - 了解项目的全局约束、禁止事项、命名规范
   - 了解项目使用的技术栈和约定

2. **读取相关子路径的 AGENTS.md**（如果有的话）
   - 了解子模块的特定约束

3. **读取相关需求/任务/计划文档**
   - 确保理解"为什么"要改，不只是"改什么"

### 第二步：读目标文件

打开你计划修改的文件，回答以下问题：

| 问题                             | 检查方式                           |
| -------------------------------- | ---------------------------------- |
| 这个文件的职责是什么？           | 看文件头部注释、函数名、导出项     |
| 有哪些导出项？它们的签名是什么？ | 看 export / public 声明            |
| 有哪些导入项？来自哪里？         | 看 import 语句，了解依赖           |
| 有哪些测试文件？测试覆盖了什么？ | 找到同名 `.test.` 或 `.spec.` 文件 |
| 文件最近有过什么变更？           | `git log --oneline -10 <file>`     |

### 第三步：读调用者与被调用者

一个文件的改动会影响谁？

| 方向                  | 检查方式                             |
| --------------------- | ------------------------------------ |
| 谁在调用这个文件？    | `grep` 搜索导入该文件的地方          |
| 这个文件调用了谁？    | 看 import 语句，阅读被调用函数的签名 |
| 有没有同名的测试？    | `grep` 搜索函数名在测试文件中的引用  |
| 类型/接口定义在哪里？ | 追溯 import 源头                     |

### 第四步：读相似实现

在代码库中找类似的功能实现：

1. **`grep` 搜索相似的功能名称**
2. **找到已存在的类似模块**，看它们是怎么实现的
3. **遵循现有模式**：新代码应该在结构、命名、错误处理上和现有代码一致
4. **不要"发明新模式"**：一致性 > 你的偏好

## 阅读深度判定

| 改动范围               | 最小阅读深度                                    |
| ---------------------- | ----------------------------------------------- |
| 修改一个函数的内部实现 | 读该函数 + 其测试 + 调用者                      |
| 新增一个函数           | 读文件全部内容 + 同级函数风格 + 测试文件        |
| 新增一个文件/模块      | 读 2-3 个同级文件 + 目录结构 + 索引文件         |
| 跨模块修改             | 读所有涉及文件的 import/export + 类型定义源     |
| 增删 API 端点          | 读路由定义 + 现有端点风格 + 中间件链 + 契约定义 |

## 常见查找命令

```
# 查找函数/类型的定义位置
grep -rn "functionName" --include="*.ts"

# 查找文件被哪些文件导入
grep -rn "from.*fileName" --include="*.ts"

# 查看文件的最近变更
git log --oneline -10 -- path/to/file

# 查看文件在某次提交前后的差异
git diff commit_hash^..commit_hash -- path/to/file

# 查找相似的实现模式
grep -rn "类似的关键词" --include="*.ts"

# 查找测试文件中对该函数的引用
grep -rn "functionName" --include="*.test.*"
```

## 常见借口

| 合理化借口                        | 现实                                                                                  |
| --------------------------------- | ------------------------------------------------------------------------------------- |
| "我大概知道这个项目怎么写的"      | 印象靠不住。特别是细节：参数顺序、错误格式、类型定义位置。必须实际读取。              |
| "改动很小，不需要看那么多"        | 改动越小越容易"我以为"——以为参数是 (a,b) 实际是 (b,a)；以为返回的是值实际是 Promise。 |
| "先写了再对，不对编译器会报错"    | 编译器报错只能发现类型问题。逻辑错误、风格不一致、重复实现——编译器不管。              |
| "这个模块我之前改过，很熟悉"      | 代码在变。你上次改是上周，别人昨天可能刚改了。重新读。                                |
| "AI 生成的代码，它应该知道怎么改" | AI 不知道你的具体代码库——除非你把它知道的喂给它。不喂 = 猜错率极高。                  |

## 红线——停下来，按流程走

如果你发现自己在想：

- "这种类型的代码我写过很多次了，不用看"
- "先写了再说，不对再改"
- "把文件打开瞄一眼就行了"
- "grep 太慢，我凭印象找文件"
- "现有代码风格不好，我按自己的来"

**以上这些都意味着：你没有在做源码驱动开发。回到第一步，实际读取代码。**
