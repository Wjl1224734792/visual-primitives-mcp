# 代码简化

## 概述

代码能跑 ≠ 代码可维护。本技能指导你在**不改变行为**的前提下降低复杂度——删死代码、提取重复、扁平化嵌套、拆分长函数、减少参数。

**核心原则：** 简化是降熵，不是加功能。每次重构必须前后行为完全一致。

**违反规则的字面意思就是违反规则的精神。**

**必需背景：** 你必须理解 `behavioral-guidelines`（准则 2：简单优先）。

## 何时使用

**适用场景：**

- 代码能跑但读起来费劲，每次改都要花很长时间理解
- 新人入职后抱怨某些模块看不懂
- 同一个 bug 修了好几次，因为代码逻辑绕来绕去
- 一个函数超过 50 行，或者嵌套超过 3 层
- 同一段逻辑在 3 个以上地方重复出现
- 函数有 5 个以上参数，每次调用都要翻定义看顺序
- 有注释写着"这段不要删，我也不知道为什么"

**不适用场景：**

- 代码很简单，只是你不熟悉业务——先读懂，别动
- 重构顺便加功能——简化就是简化，新功能分开做
- 临时代码/一次性脚本——不值当
- 没有测试覆盖的代码——先补测试，再简化

## 简化流程

必须按顺序完成每个阶段。

### 第一阶段：安全评估

**在动任何代码之前：**

1. **确认测试覆盖**
   - 运行全部测试，确保当前全部通过
   - 记录测试结果作为基线
   - 如果关键路径没有测试 → 先补测试，再简化

2. **理解代码意图**
   - 这段代码解决什么问题？
   - 谁在调用它？（grep、IDE find references）
   - 有没有隐含的边界条件？（检查 git log、相关 issue）
   - **切斯特顿之栏：** 看到不理解的东西——先弄清楚它为什么存在。不理解≠死代码。

3. **扫描简化机会**
   - 深度嵌套（>3 层）
   - 长函数（>50 行）
   - 重复逻辑（3 处以上相同/相似代码）
   - 死代码（未被引用、不可达）
   - 过度抽象（为了"将来可能"而加的抽象层）
   - 不必要的泛化（只在一个地方用的通用代码）
   - 参数过多（>5 个参数）

### 第二阶段：逐步简化

**五项简化原则，按优先级执行：**

1. **删除死代码（真正无用的，不是你不理解的）**
   - 先 grep 整个仓库确认未被引用
   - 用完 IDE find references 确认
   - 确认不是配置项、API 约定、即将上线的功能
   - 确认后：删掉。不要注释掉——Git 历史会记住它。

2. **提取重复逻辑（DRY——重复三次以上才提取）**
   - 注意：DRY 指的是**知识重复**，不是文本重复
   - 两个看起来像但含义不同的代码 → 不应合并
   - 提取前问自己：如果其中一个变了，另一个也要变吗？如果否 → 不是真正的重复

3. **扁平化深层嵌套（early return、guard clauses）**

   ```
   # 重构前：深层嵌套
   def process(order):
       if order:
           if order.is_paid:
               if not order.is_shipped:
                   ship(order)

   # 重构后：guard clauses
   def process(order):
       if not order:
           return
       if not order.is_paid:
           return
       if order.is_shipped:
           return
       ship(order)
   ```

4. **拆分过长函数（>50 行的函数考虑拆分）**
   - 每次只提取一个职责
   - 给提取的函数起清楚的名字（描述做什么，不描述怎么做）
   - 拆完后每个函数仍应可独立理解

5. **减少参数数量（>5 个参数考虑用对象封装）**
   - 将相关的多个参数组合成一个配置对象
   - 不要为了"减少参数"而引入不必要的抽象

### 第三阶段：前后验证

**每次简化后必须验证（每个小步骤，不是最后一次性验证）：**

1. 运行**完全相同的测试**，确保全部通过
2. 新增的测试也必须通过（如果补了测试）
3. 对比简化前后的测试输出——完全一致才算通过
4. 如果测试失败 → 立即回退，不要叠修复

**500 行规则：** 如果一次重构涉及的改动超过 500 行，停下来。要么拆分为多次独立的小重构，要么考虑使用自动化工具（如 IDE 的自动重构、代码迁移脚本）。

## 常见借口

| 合理化借口                    | 现实                                                                     |
| ----------------------------- | ------------------------------------------------------------------------ |
| "能跑就行，不用简化"          | 能跑的代码加上难以维护 = 技术债务，利息每天在涨。                        |
| "重构一下顺便加点功能"        | 简化 ≠ 加功能。混在一起导致 review 困难、回滚痛苦。分开做。              |
| "这个文件太乱了，我重写吧"    | 重写丢掉隐式知识（那些修了很久的边界条件）。逐步简化，保留行为。         |
| "我就是改几行，不用跑测试"    | 改一行也可能引入 bug。简化前后行为必须完全一致。跑测试。                 |
| "这个抽象看起来没用了，删掉"  | 先问历史。搜索谁在用它。确认不是配置、不是 API、不是约定——再删。         |
| "DRY 原则说不能重复"          | DRY 指的是知识重复，不是文本重复。两个看起来像但含义不同的代码不应合并。 |
| "我后面再跑测试"              | 现在就跑。拖到后面你忘了基线是什么，出了问题不知道哪个改动引起的。       |
| "只是提取一个变量，不会出错"  | "不会出错"的改动加起来就是回归 bug。每一步都要验证。                     |
| "超过 500 行就一起改了，省事" | 一大坨改动难以 review、难以回滚。拆开。每次只做一件事。                  |

## 红线——停下来，按流程走

如果你发现自己在想：

- "这段代码看起来没用，我直接删了"
- "顺便优化一下这个函数"
- "改完再一起跑测试"
- "原来的逻辑太绕了，我按自己的理解重写"
- "这个重复就两处，但也一起提取了吧"
- "代码虽然 60 行，但逻辑简单，不用拆"
- "我的简化应该没问题，测试不跑也行"
- "这块重构顺手把变量名也换成英文"
- "老代码质量太差了，不如重写"

**以上这些都意味着：停下来。回到第一阶段。**

## 验证清单

每次简化完成后，在合并或提交前确认：

- [ ] 简化前后运行了完全相同的测试，全部通过
- [ ] 测试输出与基线完全一致（无新增失败、无新增警告）
- [ ] 删除的代码已通过 grep/IDE 确认不再被引用
- [ ] 未删除任何不理解的代码（遵守切斯特顿之栏）
- [ ] 重复代码的提取确认是知识重复而非文本重复
- [ ] 单次改动不超过 500 行
- [ ] 未在简化中夹带新功能或行为变更
- [ ] 简化后的代码仍符合 `behavioral-guidelines` 准则 2（简单优先）
- [ ] 嵌套层级不超过 3 层
- [ ] 无函数超过 50 行
- [ ] 无函数超过 5 个参数

不能全部勾选？回退，按流程重新来。
