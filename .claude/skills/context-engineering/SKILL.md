# 上下文工程

## 概述

上下文工程是将正确信息在正确时机传递给智能体的系统化方法。它不是"多传点总是好的"——上下文是一种稀缺资源，过量与不足同样有害。

**核心原则：** 在每一个决策点，给智能体恰好足够的信息做出好决策——不多一行，不少一行。

**此技能是 behavioral-guidelines 的配套技能。** behavioral-guidelines 规定智能体"怎么做事"，context-engineering 规定它"知道什么"。

---

## 何时使用

**必须使用的场景：**

- 启动新会话（无论是编排会话还是子智能体会话）
- 任务切换——从一个功能跳转到另一个不相关的功能
- 智能体输出质量明显下降——开始生成假方法、忽略约束、或偏离项目模式
- 子智能体回传的结果与预期严重偏离
- 在长时间会话中进行"上下文重置"

**不应使用的场景：**

- 智能体已在当前任务上正常工作且输出质量良好——不要为了"优化"打断它
- 当前任务只是上一个任务的微小延续——上下文天然连贯
- 排查同一个 bug 的连续迭代——上下文应该累积，而非重置

## 上下文五级层次

理解不同层次的上下文何时加载、为何加载：

```
第 5 级：对话历史          ← 累积，需压缩
第 4 级：错误/测试输出      ← 按迭代加载
第 3 级：相关源码            ← 按任务加载
第 2 级：规格/架构文档      ← 按功能加载
第 1 级：规则文件            ← 始终加载
```

### 第 1 级：规则文件（始终加载）

AGENTS.md、CLAUDE.md、项目级 agent .md 文件。定义项目级约束——构建命令、lint 规则、代码风格、测试框架、目录约定。

**原则：** 这是上下文地基。缺失或错误 → 所有后续决策都建在沙子上。

### 第 2 级：规格/架构文档（按功能加载）

需求文档、设计规格、ADR（架构决策记录）。告诉智能体"要建什么"和"为什么这样设计"。

**原则：** 只在当前功能相关时加载。不要为一个按钮 color fix 加载整个微服务架构 ADR。

### 第 3 级：相关源码（按任务加载）

当前任务涉及的文件、调用链、接口契约、相邻模块的类型定义。

**原则：** 这是上下文预算消耗最大的层级。选择性注入是关键——目标是 <2000 行有效上下文。

### 第 4 级：错误输出/测试结果（按迭代加载）

编译器错误、测试失败栈、lint 警告。每一次修复迭代都应注入新鲜的错误/测试输出。

**原则：** 只传错误摘要 = 丢失根因。传完整输出，包括上下文行号。

### 第 5 级：对话历史（累积，需压缩）

之前轮次的澄清、决策、被否决的方案。

**原则：** 积累太多 → 上下文洪泛。当对话历史超过当前任务相关上下文时，应进行分层压缩。

## 过程

### 步骤 1：评估当前上下文状态

启动会话或切换任务时，先判断：

1. **智能体手头有什么？** 当前加载了哪些 1-5 级上下文？
2. **任务需要什么？** 哪些层次是必需的？
3. **缺口是什么？** 缺了哪个层级的内容？

**输出：** 一个隐含的"上下文清单"——缺什么补什么，多余的标记为潜在洪泛来源。

### 步骤 2：选择打包策略

根据任务范围，选择以下策略之一：

#### 策略 A：脑力倾卸（Brain Dump）——会话启动时

适用场景：全新会话开始，智能体零上下文。

**打包内容：**

```markdown
## 项目背景

[一句话描述项目目的和当前阶段]

## 相关规格

[当前功能的设计规格摘要，或引用具体文档]

## 关键约束

[框架、语言、构建工具、测试命令、不可触碰的区域]

## 涉及文件

[粗略文件列表或目录映射，标注关键入口点]

## 应遵循的模式

[具体引用的现有实现、约定、命名规范]

## 已知陷阱

[之前踩过的坑——竞态、状态管理、类型对不齐等]
```

**原则：** 这是一个初始化负载。不应超过当前会话上下文预算的 30%。

#### 策略 B：选择性注入（Selective Include）——任务执行中

适用场景：范围明确的小任务、快速修复。

**做法：**

- 只传当前任务直接相关的文件和文档
- 目标：有效上下文 <2000 行
- 明确排除不相关的内容："以下文件仅供参考，不需要修改"

**反模式：** "我把整个 repo 都传进去"——智能体会在无关文件间跳来跳去。

#### 策略 C：分层摘要（Hierarchical Summary）——大型项目

适用场景：代码库庞大、跨模块任务。

**做法：**

- 提供项目地图/索引：列出模块及其职责
- 每个模块标注关键文件和核心模式
- 只展开当前任务相关的模块章节
- 未展开的章节保留为引用锚点——需要时再深入

**原则：** 索引级描述 ≈100 行 → 模块级展开 ≈500 行 → 文件级深入 ≈按需。

#### Jarvis 适配说明

当前 Jarvis（编排者）要求"传递完整上下文文档"。这是**基准线**——不是上限。对于明确范围的小任务：

- 基准线上下文（AGENTS.md、项目约定）必须包含
- 额外上下文使用选择性注入
- 目的：减少 token 消耗，同时在 Gate 检查时确保子智能体拿到足够信息

### 步骤 3：管理混淆

上下文冲突时，遵循三原则：

#### 原则 1：直接暴露矛盾

如果规格文档与现有代码矛盾：

**正确：** "规格说 REST 接口，但现有代码用 GraphQL——用哪个？"
**错误：** 默默选一个并开始实现。

#### 原则 2：需求不完整时先查后问

1. 查对话历史——之前的轮次是否澄清过？
2. 查规格文档——设计文档中是否有说明？
3. 如果两者都没有——停下来，向用户提出具体问题。
4. **绝不凭空发明需求。**

#### 原则 3：使用内联规划模式（Inline Planning Pattern）

对于多步骤任务，在执行前输出轻量计划：

```
PLAN:
1. 添加 Zod schema → 验证: TypeScript 编译通过
2. 接入路由 handler → 验证: curl 测试 200
3. 添加单元测试 → 验证: npm test 通过
→ 除非你纠正，否则我将按此执行。
```

**目的：** 给用户在智能体走错方向之前纠正的机会，同时不消耗一个完整的规划周期。

### 步骤 4：识别和处置反模式

当你观察到以下症状时，说明上下文出了问题：

| 反模式         | 症状                                                   | 处置                                                       |
| -------------- | ------------------------------------------------------ | ---------------------------------------------------------- |
| **上下文饥饿** | 智能体 hallucinates——生成不存在的 API、发明工具        | 补充第 3 级（相关源码），提供接口契约和调用示例            |
| **上下文泛滥** | 智能体丢失焦点——在不同文件间无意义跳转，输出与任务无关 | 裁剪上下文至 <2000 行相关行，使用选择性注入                |
| **过期上下文** | 智能体引用已删除的代码、使用旧模式                     | 更新第 1-2 级上下文，刷新文件映射                          |
| **缺少示例**   | 智能体发明新风格而非遵循项目约定                       | 注入正面示例：具体的、现有的正确实现文件                   |
| **隐式知识**   | 智能体忽略项目约定——因为没人写出来                     | 将隐式知识写入 AGENTS.md 或任务打包中。没写下来的 = 不存在 |
| **沉默混淆**   | 智能体猜测而非提问——输出有微妙错误                     | 在任务打包中加上："如果不确定 X，停下来问。不要猜。"       |

## 常见合理化借口

| 合理化借口                           | 现实                                                                                  |
| ------------------------------------ | ------------------------------------------------------------------------------------- |
| "多传点上下文总没坏处"               | 上下文洪泛导致智能体丢失焦点。更多上下文 ≠ 更好输出。超过 5000 行时输出质量开始下降。 |
| "我把整个 repo 都传进去"             | 智能体会在无关文件间跳来跳去。"全量"上下文让智能体分心。                              |
| "我记得项目结构，不用查"             | 记忆会过时。文件改了、模式变了，你以为的东西已经不对了。                              |
| "错误信息我看懂了，不用传全量"       | 只传部分错误 = 丢失根因。传完整错误堆栈，包括上下文行号。                             |
| "上下文不够时智能体会问的"           | 智能体宁可猜测也不愿显得无知。你不给够，它就编。                                      |
| "系统自动加载了 AGENTS.md，我不用管" | AGENTS.md 只是第 1 级。第 2-4 级取决于你注入什么。                                    |

## 红线——停下并修复上下文

以下信号表示上下文工程出了问题，应立即处理：

- 智能体生成的 API/函数/类型在代码库中不存在（上下文饥饿）
- 智能体在 3 个以上不相关文件间跳来跳去没有产出（上下文泛滥）
- 智能体引用 2 周前已删除的模块或模式（过期上下文）
- 智能体的代码风格与项目约定完全不同（缺少示例）
- 智能体的输出忽好忽坏——对 A 部分正确，B 部分完全是编的（上下文不均）
- 子智能体回传的 Execution Acknowledgement 包含对任务的明显误解
- 同一个问题上与智能体反复澄清超过 3 轮——信息没传进去

## 验证清单

在宣称"上下文已就绪"之前，逐项确认：

- [ ] 第 1 级（规则文件）：AGENTS.md 及项目级约束已加载，且是最新版本
- [ ] 第 2 级（规格/架构）：当前功能相关设计文档已注入或已引用
- [ ] 第 3 级（相关源码）：任务涉及的核心文件已提供，总行数 <2000（或使用分层摘要展开相关章节）
- [ ] 第 4 级（错误/测试）：如有失败输出，已提供完整栈（含行号和上下文）
- [ ] 第 5 级（对话历史）：历史决策已压缩为摘要（而非完整对话），遗留的隐式知识已显式化
- [ ] 打包策略：已根据任务范围选择合适的策略（脑力倾卸 / 选择性注入 / 分层摘要）
- [ ] 混淆检查：不存在规格与代码矛盾、需求缺口、或未暴露的歧义
- [ ] 示例检查：如需遵循特定代码模式，已提供一个具体的正面示例（文件路径 ＋ 行号范围）
- [ ] 反模式扫描：无饥饿、无泛滥、无过期、无隐式知识

**以上全部确认后，上下文打包才算完成。**
