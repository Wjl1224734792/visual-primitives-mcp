# 完成前验证

## 概述

"做完了"是一句危险的话。太多 bug 来自于"我以为完成了但没有验证"。本技能提供一套标准化的完成前检查清单，确保在宣称完成之前，实现真的通过了所有验收标准。

**核心原则：** 未经验证的完成 = 未完成。测试通过 ≠ 全部完成。代码写完 ≠ 可以交付。

**必需背景：** 你必须理解 `behavioral-guidelines`（准则 4：目标驱动执行）和 `test-driven-development`。

## 何时使用

**适用场景：**

- 每个实现子任务交付前（强制）
- 每个 Bug 修复完成后
- 每个功能迭代完成后
- 向编排者 回报"任务完成"之前

**不适用场景：**

- 纯探索/研究任务（但探索结果的质量仍需验证）
- 中间步骤的临时保存（commit 不等于完成）
- 被明确豁免验证的任务（需记录豁免原因）

## 验证清单

### 第一层：功能正确性

在"完成"之前，确认：

- [ ] **代码实现了所有验收标准**：对照 Execution Packet 或需求文档逐条检查
- [ ] **正向路径通过**：正常输入产生正常输出
- [ ] **边界条件覆盖**：空输入、最大值、最小值、null/undefined
- [ ] **错误路径处理**：无效输入返回明确错误，不崩溃不吞异常
- [ ] **无竞态条件**：如果涉及异步/并发，确认状态转换正确
- [ ] **幂等性（如适用）**：重复执行同一操作不会产生副作用
- [ ] **数据一致性**：写入的数据可以被正确读取

### 第二层：自动化验证

- [ ] **lint 通过**：运行项目的 lint 命令，零新增警告/错误
- [ ] **typecheck 通过**：运行项目的类型检查命令，零类型错误
- [ ] **单元测试通过**：所有新增和现有的单元测试通过
- [ ] **集成测试通过**（如果涉及）：API 测试、数据库测试等通过
- [ ] **E2E 测试通过**（如果涉及）：关键用户旅程测试通过
- [ ] **构建成功**（如果涉及）：项目可成功构建，无构建错误

### 第三层：非功能验证

- [ ] **无安全漏洞**：快速自查 OWASP 前十（XSS、SQL 注入、密钥泄露等）
- [ ] **性能可接受**：无明显 N+1 查询、无限循环、同步阻塞
- [ ] **无障碍（a11y）**（前端）：基本可访问性（语义标签、键盘操作、颜色对比度）
- [ ] **响应式（前端）**：在目标屏幕尺寸下布局正常
- [ ] **错误消息清晰**：用户看到的错误信息是可理解的（不是栈追踪）
- [ ] **日志适当**：关键路径有日志，但不泄露敏感信息

### 第四层：交付物完整

- [ ] **代码已提交**：变更已通过 `git` 提交（如果适用）
- [ ] **实现文档已输出**：如果有要求，`docs/implementation/` 下的文档已写入
- [ ] **测试证据保留**：关键测试运行输出已留档
- [ ] **变更范围可控**：未超出 Execution Packet 的 `allowed_paths`
- [ ] **无遗留调试代码**：console.log、临时注释、硬编码的测试数据已清理
- [ ] **TASK-XXX 追溯完整**：改动可追溯到对应的任务和需求

### 第五层：上下文清理

- [ ] **数据库迁移可回滚**（如果涉及）：up 和 down 脚本都存在且正确
- [ ] **特性开关配置**（如果涉及）：新增功能的特性开关已定义
- [ ] **依赖已声明**（如果涉及）：新引入的库已添加到 `package.json` 等文件
- [ ] **环境变量文档**（如果涉及）：新增的环境变量已记录
- [ ] **API 契约已更新**（如果涉及）：新增/变更的 API 契约已记录

## 验证失败的处理

如果验证未通过，按优先级处理：

| 失败类型                | 处理方式                                 |
| ----------------------- | ---------------------------------------- |
| 功能验收标准未满足      | **最高优先级**——继续实现，不可交付       |
| lint/typecheck 失败     | 修复后重新验证，不可跳过                 |
| 测试失败                | 逐个分析：是测试错了还是代码错了？       |
| 非功能问题（性能/安全） | 低风险可标记为已知风险；中高风险必须修复 |
| 交付物缺失（文档/日志） | 补齐后重新验证                           |

## 快速自检（30 秒版）

在时间紧张时至少确认这 5 条：

1. 💚 **测试绿吗？** 新增和现有测试全部通过
2. 🔍 **lint 过吗？** 无新增警告或错误
3. 📋 **验收标准到了吗？** 对照 checklist 逐条勾过
4. 🧹 **清理了吗？** 无 console.log、注释掉的代码、硬编码测试数据
5. 📏 **范围对吗？** 没改不该改的文件

## 常见借口

| 合理化借口                         | 现实                                                                      |
| ---------------------------------- | ------------------------------------------------------------------------- |
| "测试过了，肯定没问题"             | 测试过的是你写的测试用例。如果测试用例本身就没覆盖边界条件呢？            |
| "lint 报的都是旧代码的错"          | 那至少保证你的新代码没增加新的 lint 错误。修复旧 lint 错误是独立任务。    |
| "就改了一行，不用跑全量测试"       | 一行代码可能导致类型错误、运行时异常、安全漏洞。至少跑相关测试。          |
| "功能做完了，文档后面补"           | 不写文档 = 没完成。其他开发者怎么理解你的代码？3 个月后的你怎么理解？     |
| "验证太费时间了"                   | 验证费 5 分钟。不验证导致的返工和排查费 5 小时。                          |
| "console.log 留着没关系，方便调试" | 生产环境的 console.log 可能泄露敏感信息、影响性能。删掉。需要调试时再加。 |

## 红线——停下来，按流程走

如果你发现自己在想：

- "差不多了，应该没问题"
- "这条验证项不太重要，跳过"
- "测试失败了，应该是我测试写错了，代码没问题"
- "lint 没过，但这不是我改的文件导致的"
- "文档内容很清晰，不写也行"

**以上这些都意味着：验证不完整。逐条过检查清单。**

## 验证清单（汇总）

完成后逐条勾选：

**功能正确性：**

- [ ] 所有验收标准已满足
- [ ] 正向路径通过
- [ ] 边界条件覆盖
- [ ] 错误路径处理正确
- [ ] 无竞态条件（如适用）

**自动化验证：**

- [ ] lint 通过（零新增错误）
- [ ] typecheck 通过（零类型错误）
- [ ] 单元测试通过
- [ ] 集成测试通过（如涉及）
- [ ] 构建成功（如涉及）

**非功能验证：**

- [ ] 无明显安全漏洞
- [ ] 性能可接受
- [ ] 错误消息清晰

**交付物完整：**

- [ ] 实现文档已输出
- [ ] 测试证据已保留
- [ ] 无遗留调试代码
- [ ] 变更范围未越界
- [ ] TASK-XXX 追溯完整

**上下文清理：**

- [ ] 数据库迁移可回滚（如涉及）
- [ ] 依赖已声明（如涉及）
- [ ] API 契约已更新（如涉及）
