# 文档与架构决策记录

## 概述

好文档决定团队的长期效率。好文档不是写得最全的，而是**对的人、在对的时间、能迅速找到对的信息**。坏的文档比没有文档更糟——它会让人浪费时间读完后发现过时了、不相关、或者完全看不懂。

本技能覆盖三种文档类型：

| 文档类型                | 记录什么                     | 读者                 |
| ----------------------- | ---------------------------- | -------------------- |
| **ADR（架构决策记录）** | 为什么做了这个架构选择       | 当前和未来的团队成员 |
| **API 文档**            | 接口契约——怎么调用、返回什么 | 前后端、第三方集成方 |
| **内联文档**            | 代码里的注释——为什么这么写   | 维护这段代码的人     |

**核心原则：文档记录「为什么」，代码说明「是什么」。**

**写作规范：** 所有中文文档必须遵循 `chinese-documentation` 技能中的排版、术语、中英混排规范。下文不再重复这些规则，请自行加载该技能并在写作时逐条检查。

## 何时使用本技能 / 何时不使用

### 使用本技能（写文档）

- **选择技术栈**：选择数据库、消息队列、缓存方案、编程语言版本
- **引入新依赖**：引入重量级或决策性的第三方库（如 ORM 框架、状态管理方案）
- **重大架构变更**：服务拆分、单体转微服务、数据结构重构
- **数据模型设计决策**：表结构设计、索引策略、数据分片方案
- **API 契约变更**：公共接口的参数、返回值、鉴权方式变更
- **功能交付**：新模块上线前补齐接口文档和 README
- **系统交接或新人入职**：确保知识不只在某个人的脑子里

### 不使用本技能（不写文档）

- **小改动的实现细节**：内部 helper 函数、单个字段改名、局部重构
- **可轻易反转的选择**：换个配置、换种写法，成本极低
- **个人风格偏好**：括号换行、命名风格（统一用 ESLint/Prettier 管）
- **"我们一直这么做"**：没有经过思考的惯例不值得记录
- **临时代码/原型**：明确会丢弃的实验性代码

## 流程

### 步骤 1：判断是否值得写

做任何技术决策前，先问自己三个问题：

1. **这个决定半年后还需要解释吗？** 如果不需要，不写。
2. **新人看到这段代码会困惑吗？** 如果会，写注释。
3. **其他人调用这个接口需要猜测参数含义吗？** 如果会，写 API 文档。

### 步骤 2：选择文档类型

根据上方的使用场景表，确定要写哪种文档。一个决策可能需要多种文档配合——例如引入新数据库时，同时写一个 ADR 记录选型过程，并更新 API 文档中相关接口的说明。

### 步骤 3：写 ADR（如适用）

使用下方 ADR 模板，按以下顺序写：

1. **Title**：动词 + 对象 + 目的，如「使用 PostgreSQL 作为主数据库」
2. **Status**：当前状态（Proposed → Accepted → Deprecated → Superseded）
3. **Context**：当时的背景——面临什么问题、有什么约束、谁提出来的
4. **Decision**：做了什么决定，写清楚具体方案
5. **Consequences**：这个决定带来的正面和负面影响
6. **Alternatives Considered**：考虑过的其他方案及为什么没选

### 步骤 4：写 API 文档（如适用）

参考本技能 API 文档部分的标准模板，重点确保：

- 请求/响应示例可以直接复制执行
- 每个字段有中英对照说明
- 错误码有明确的含义和处理建议
- 鉴权方式写清楚（Header 名称、Token 获取方式）

### 步骤 5：写代码注释（如适用）

对于关键逻辑、非显而易见的实现、hack/workaround 写法，写注释说明原因：

```
# 好：说明为什么这样做
# 使用二分查找而非哈希表，因为数据集在运行时动态增长且需要范围查询

# 坏：重复代码内容
# 遍历数组
for item in items:
    # 处理每个 item
    process(item)
```

### 步骤 6：自检

对着本技能末尾的验证清单逐项检查。确保文档可读、可执行、没有机翻味。

### 步骤 7：提交与维护

- 把文档放在代码仓库里，和代码一起进行版本管理
- ADR 放在 `docs/adr/` 目录下
- 在 PR 中引用相关 ADR，让 reviewer 也看到决策背景
- 每次修改模块时顺手检查并更新相关文档

## ADR 模板

```markdown
# NNNN — 决策标题

**状态 (Status):** Proposed | Accepted | Deprecated | Superseded by [ADR-NNNN](./NNNN-xxx.md)

**日期 (Date):** YYYY-MM-DD

**决策者 (Deciders):** @某人, @某人

## 上下文 (Context)

描述当前面临的技术问题、业务背景和约束条件。
为什么到了必须做决定的时候？

例如：

- 当前系统使用内存缓存，但重启后数据丢失，导致用户登录状态不一致
- 用户量增长到 10 万，原有缓存方案无法水平扩展
- 运维要求数据持久化，减少重启后的错误率

## 决策 (Decision)

明确写出我们做了什么决定。

例如：
我们将引入 Redis 作为集中式缓存层：

- 版本：Redis 7.x
- 部署方式：主从 + Sentinel 高可用
- 序列化方案：JSON（兼容现有数据结构）
- 缓存策略：Cache-Aside（先查缓存，未命中再查 DB）
- 过期策略：用户 Session 的 TTL 为 30 分钟

## 后果 (Consequences)

### 正面影响

- 重启不再丢失数据，用户无需重新登录
- 可水平扩展，满足未来 3 年的增长需求
- 所有服务共享同一缓存，消除数据不一致

### 负面影响

- 引入新的运维依赖，需要监控 Redis 集群健康状态
- 网络延迟增加约 1-2 ms
- 多一层故障点——Redis 不可用时需要降级方案

### 缓解措施

- 实现 circuit breaker，Redis 不可用时自动降级为本地缓存
- 添加 Redis 监控告警，关键指标：内存使用率、连接数、延迟 P99
- 运维团队已完成 Redis 部署培训

## 考虑的替代方案 (Alternatives Considered)

### 方案 A：Memcached

- **优点：** 比 Redis 更轻量，单线程模型简单
- **缺点：** 不支持持久化，数据结构单一，团队缺乏运维经验
- **弃用原因：** 需要持久化能力，Redis 的数据结构更丰富符合需求

### 方案 B：应用层本地缓存 + 数据库

- **优点：** 不引入新组件，架构简单
- **缺点：** 多实例间数据不一致，无法共享缓存
- **弃用原因：** 微服务架构下数据一致性是硬需求
```

## ADR 命名规则

ADR 文件命名遵循以下规则：

```
NNNN-title-with-dashes.md
```

| 规则                 | 说明                  | 示例                                |
| -------------------- | --------------------- | ----------------------------------- |
| 起始四位数编号       | 从 0001 开始递增      | `0001`                              |
| 标题用英文小写连字符 | 方便 URL 和命令行补全 | `use-postgres-for-persistence`      |
| 不要中文文件名       | 编码问题和跨平台兼容  | `引入-PostgreSQL-做主库.md`（禁止） |

**正确示例：**

```
docs/adr/0001-use-postgres-for-persistence.md
docs/adr/0002-adopt-redis-as-cache-layer.md
docs/adr/0003-migrate-monolith-to-microservices.md
docs/adr/0004-deprecate-legacy-v1-api.md
```

**错误示例：**

```
docs/adr/使用Redis做缓存.md          # 中文文件名
docs/adr/1-redis.md                   # 编号不足 4 位
docs/adr/0001_use_redis_as_cache.md   # 使用下划线而非连字符
docs/adr/redis-decision.md            # 没有编号
```

## ADR 状态流转

```
Proposed ──→ Accepted ──→ Deprecated
                │
                └──→ Superseded by ADR-NNNN
```

| 状态           | 含义                 | 何时进入                         |
| -------------- | -------------------- | -------------------------------- |
| **Proposed**   | 提议中，等待团队讨论 | 初始提交，创建 PR                |
| **Accepted**   | 已通过，正在执行     | PR 合入主分支                    |
| **Deprecated** | 不再适用             | 这不再是我们遵循的决策           |
| **Superseded** | 被新 ADR 替代        | 新的决策覆盖了这个，标注替代关系 |

**注意：** 不要删除旧的 ADR。它们是项目演进的历史档案。用 `Deprecated` 或 `Superseded` 标记即可。

## API 文档标准

API 文档详细模板和排版规范请参考 `chinese-documentation` 技能，本节只补充额外标准。

### 强制内容

每份 API 文档必须包含以下 8 项：

1. **基本信息**：Method、Path、Content-Type、鉴权方式
2. **请求参数**：字段名、类型、必填/选填、说明（中英对照）
3. **请求示例**：可直接复制的完整示例
4. **响应参数**：字段名、类型、说明
5. **成功响应示例**：完整的 JSON 响应
6. **错误响应示例**：至少一个典型错误场景
7. **错误码**：code、说明、处理建议
8. **变更记录**：版本、日期、变更内容

### 字段说明要求

每个字段的说明必须包含：

```
# 好
| user_id | string | 用户 ID（User ID），由系统在注册时自动生成，格式为 uuid v4 |

# 坏
| user_id | string | 用户 ID |
```

- 业务含义：这个字段代表什么
- 格式约束：长度、枚举值、正则、特殊格式
- 来源：前端传入 / 系统生成 / 第三方返回

### 金额与数值约定

- 金额字段使用最小单位（分），避免浮点精度问题，并在文档中明确标注
- 时间字段统一使用 ISO 8601 格式，注明时区
- 枚举值列出所有合法取值

### 鉴权文档标准

```
## 鉴权 (Authentication)

本接口使用 Bearer Token 鉴权。

### 获取 Token

调用 POST /api/v1/auth/login 获取 Token，详见 [登录接口文档](./api-login.md)。

### 使用方式

在请求 Header 中携带：
Authorization: Bearer <your-token>

### Token 有效期

- Access Token：2 小时
- Refresh Token：7 天

过期后使用 Refresh Token 续期，详见 [Token 续期文档](./api-token-refresh.md)。
```

## 内联代码文档

### 注释原则

| 注释写什么                   | 不写什么                             |
| ---------------------------- | ------------------------------------ |
| 为什么选择这个算法而非另一个 | 这段代码在做什么（代码本身能看出来） |
| 为什么这里用了 workaround    | 逐行翻译代码                         |
| 性能敏感的假设               | 显而易见的赋值和函数调用             |
| 非直觉的边界条件             | 过期的 TODO（会被 IDE 高亮但不解决） |
| 魔法数字的来源（公式/参考）  | 个人签名或表情符号                   |

### 示例

```python
# 好：说明算法选择原因
# 此处使用 Rabin-Karp 而非 KMP。数据集以短模式串为主，
# Rabin-Karp 的哈希滚动特性使得多模式匹配只需一次遍历，比 KMP 平均快 3 倍。
# 见 benchmark: benchmark/algos/string-match/result.md

# 好：说明 workaround 原因
# NOTE: Django 3.2 的 select_related() 在跨数据库关联时存在 N+1 查询 bug。
# 升级到 Django 4.x 后可移除此 workaround。详见 https://code.djangoproject.com/ticket/33361
queryset = Model.objects.raw('SELECT ...')

# 好：说明边界条件
# ID 为 0 时表示系统根节点，不可删除。该记录在数据迁移中手动创建。
if node_id == 0:
    raise CannotDeleteRootNode()

# 坏：重复代码内容
# 查询所有用户
users = User.objects.all()

# 坏：过期的 TODO
# TODO: 以后优化这里  ← 这是 2019 年的 TODO，永远不会做了

# 坏：无意义的注释
# 把 i 加 1
i = i + 1
```

### 注释语言

- **项目开发语言为中文团队**：注释用中文，便于项目内所有人阅读
- **开源项目 / 国际化团队**：注释用英文
- **外部 API 的文档字符串**：推荐中英双语（如 Rust 的 `///` 或 Python 的 docstring）

## README 结构

### README 强制包含

1. **一句话简介**：这个项目是什么、解决什么问题

```markdown
# 更新日志

本文档遵循 [Keep a Changelog](https://keepachangelog.com/zh-CN/1.1.0/) 规范。

## [1.2.0] - 2024-03-15

- 订单列表接口改为分页返回，默认每页 20 条
- 数据库连接池上限从 50 调整到 100

### 废弃 (Deprecated)

- `GET /api/v1/users/:id/avatar` 已废弃，请改用 `GET /api/v2/users/:id/profile` 中的 avatar 字段

### 修复 (Fixed)

- 修复并发下单导致的库存超卖问题（详见 ADR-0003）
- 修复微信支付回调在特殊字符下签名验证失败的问题

### 安全 (Security)

- 升级 jackson-databind 至 2.16.1，修复 CVE-2023-35116
```

### 规则

- 日期格式：`YYYY-MM-DD`
- 版本号遵循 [语义化版本](https://semver.org/lang/zh-CN/)
- 分类必须使用以上五种：新增、变更、废弃、修复、安全
- 每次发布必须更新 CHANGELOG，不得事后补写
- CHANGELOG 放在仓库根目录，和代码一起发版

## 常见合理化借口

写文档最大的敌人不是时间，而是自我合理化的惰性。识别并驳斥以下借口：

| 合理化借口                  | 现实                                                                                       |
| --------------------------- | ------------------------------------------------------------------------------------------ |
| 「代码自解释，不需要文档」  | 代码说 WHAT，不说 WHY。为什么选这个算法、为什么弃用那个方案——代码里看不出来。              |
| 「写文档太花时间」          | 写下来团队省时间。别人不需要追问你，你自己 6 个月后也记得。                                |
| 「ADRs 太重了，我们用不上」 | ADR 就是几段话。一个轻量的决策记录比凭记忆做决策好得多。                                   |
| 「等我做完再补文档」        | 「做完」永远不会来。边做边写，每次改动顺手更新。                                           |
| 「接口文档后端写就行」      | 前端也需要知道契约。缺少文档 = 猜测 = Bug。                                                |
| 「README 有就行」           | 坏的 README 比没有更糟。新人看完还不知道怎么跑起来。                                       |
| 「文档过时了容易误导人」    | 没错，所以文档要随代码一起更新。过时文档旁边标注「最后更新」日期，或用 `Deprecated` 标记。 |
| 「团队的人都知道，不用写」  | 新成员不知道。转岗的同学不知道。三年后的你也会忘记。                                       |

## 危险信号

以下迹象说明文档出了问题，需要立即纠正：

- **ADR 超过 500 字**：太啰嗦了。ADR 是决策摘要，不是设计文档。如果需要详细设计，额外写一份 Readme 或设计文档，ADR 中放链接。
- **API 文档中没有错误码**：调用方遇到错误只能靠猜。每个接口至少列出 3 个常见错误码及其处理建议。
- **注释里大量「修复 xxx bug」**：说明你在注释里写日记。Bug 修复应该在 commit message 里，不在注释里。
- **README 里没有「快速开始」**：这就是很多项目没人用的原因。新来的人看了三段文字还跑不起来就走了。
- **CHANGELOG 里写「修复若干 bug」**：这种 CHANGELOG 没有信息量。具体列出每条修复对应的 Issue 编号。
- **ADR 全是 Accepted，没有 Deprecated**：说明团队没有在演进。旧的决策应该被新的替代，旧 ADR 标记为 Deprecated 或 Superseded。
- **几个人各写一套格式**：文档不像文档，像聊天记录。建立模板并强制执行。
- **文档和代码不一致**：代码改了一周了，文档还是旧的。这是代码审查的问题——review 时应该要求同步更新文档。
