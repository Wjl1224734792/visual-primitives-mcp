# 代码审查与质量

## 概述

代码审查不是找茬，是**质量保证的最后一道防线**。本技能提供结构化的审查方法论，确保每段代码在合并前都经过多维度评估。

**核心原则：** 审查检查的是代码质量，不是作者的偏好。每条 finding 必须有证据支撑（文件路径、行号、命令输出）。

**必需背景：** 你必须理解 `behavioral-guidelines`（准则 3：精准修改）。

## 何时使用

**适用场景：**

- PR / diff 代码审查
- 实现交付后的质量评审
- 代码重构后的安全审查
- 新人代码的导师审查
- 合并前的最后检查

**不适用场景：**

- 项目级架构审查（交给 `project-review-expert`）
- 性能专项审查（交给 `perf-review-expert`）
- 审查范围超过 1000 行变更（建议拆分审查）

## 五轴审查框架

每个变更至少从以下五个维度评估：

### 1. 正确性

| 检查项             | 问题示例                                   |
| ------------------ | ------------------------------------------ |
| 是否符合需求规格？ | 有没有遗漏验收标准？                       |
| 边界条件是否处理？ | 空值、最大值、负数、零值                   |
| 错误路径是否覆盖？ | 异常是否被正确处理？有没有空的 catch 块？  |
| 是否存在竞态条件？ | 异步操作是否考虑了并发？                   |
| 数据是否一致？     | 写入的数据能否正确读取？事务边界是否合理？ |
| 是否幂等？         | 重复执行是否产生副作用？                   |

### 2. 可读性与简洁性

| 检查项             | 问题示例                               |
| ------------------ | -------------------------------------- |
| 命名是否清晰？     | 函数名描述"做什么"而非"怎么做"         |
| 控制流是否直观？   | 深层嵌套、过多分支、goto 式跳转        |
| 是否能用更少代码？ | 冗余变量、不必要的中间步骤、重复逻辑   |
| 逻辑是否自解释？   | 需要注释解释的代码通常是不够清晰的代码 |
| 文件职责是否单一？ | 一个文件做了太多不相关的事             |

### 3. 架构

| 检查项             | 问题示例                         |
| ------------------ | -------------------------------- |
| 是否契合系统设计？ | 新代码是否违反了现有的分层约定？ |
| 模块边界是否清楚？ | 跨层调用、循环依赖、紧耦合       |
| 抽象层级是否适当？ | 为单点使用创建过度抽象？         |
| 依赖方向是否合理？ | 底层模块依赖了上层模块？         |

### 4. 安全

| 检查项                   | 问题示例                       |
| ------------------------ | ------------------------------ |
| 输入是否验证？           | SQL 注入、XSS、命令注入风险    |
| 密钥是否安全？           | 硬编码密钥、密钥在日志中出现   |
| 鉴权是否到位？           | 缺少权限检查、可被越权访问     |
| 查询是否参数化？         | 拼接 SQL、拼接命令             |
| 外部数据是否视为不可信？ | 前端传来的数据直接用于后端逻辑 |

### 5. 性能

| 检查项                 | 问题示例                       |
| ---------------------- | ------------------------------ |
| 是否存在 N+1 查询？    | 循环内发起数据库查询           |
| 是否有无限循环风险？   | while(true) 缺少退出条件       |
| 同步操作是否应为异步？ | 阻塞主线程的同步 I/O           |
| 是否缺少分页？         | 全量查询可能返回百万条记录     |
| 是否存在内存泄漏？     | 事件监听未清理、闭包持有大对象 |

## 变更规模标准

| 变更行数    | 评估   | 操作                                         |
| ----------- | ------ | -------------------------------------------- |
| ≤100 行     | 良好   | 一次审查可完成                               |
| 100-300 行  | 可接受 | 需是一次逻辑变更（不能是多个不相关改动拼凑） |
| 300-1000 行 | 偏大   | 建议拆分为多次独立审查                       |
| >1000 行    | 过大   | 要求拆分，否则审查质量无法保证               |

---

## 严重度分级

| 标签             | 含义     | 作者操作                               |
| ---------------- | -------- | -------------------------------------- |
| **[CRITICAL]**   | 阻塞合并 | 必须修复：安全漏洞、数据丢失、功能破坏 |
| **[IMPORTANT]**  | 应修复   | 必须修复：潜在 bug、架构问题、测试缺口 |
| **[SUGGESTION]** | 建议改进 | 值得考虑但不必强制修复                 |
| **[NIT]**        | 轻微偏好 | 作者可忽略的风格偏好                   |

**注意：** 不要把风格偏好（NIT）伪装成缺陷（IMPORTANT）。只有影响正确性、安全性、可维护性的才算缺陷。

## 依赖审查

引入新依赖时检查：

1. **必要性**：现有技术栈能否解决？是否为了一个函数引入整个库？
2. **体积**：对 bundle size 有多大影响？
3. **活跃度**：最近 3 个月有 commit 吗？issue 有人回复吗？
4. **安全性**：`npm audit` / `pip audit` 是否有已知漏洞？
5. **许可证**：是否与项目许可证兼容？

## 死代码处理

发现疑似死代码时的标准做法：

```
⚠️ 疑似死代码：
- function legacyFormat() in src/utils/date.ts
  → 搜索整个仓库无引用
  → 请确认：这些可以安全删除吗？
```

**不要直接删除。** 先报告，让作者确认。可能是：

- 即将上线的功能预留
- 配置项/约定的一部分
- 外部脚本的依赖

## 审查输出格式

```
## Findings
- [P1/CRITICAL] 标题
  文件: path:line
  证据: <具体代码片段或命令输出>
  影响: <对系统的影响>
  建议: <修复建议>

- [P2/IMPORTANT] 标题
  ...

## Open Questions
<需要作者澄清的问题>

## Test Gaps
<未覆盖的测试场景>

## Residual Risk
<审查后仍存在的风险>
```

没有发现问题时，明确写"未发现阻塞问题"，并列出未覆盖的验证范围。

## 常见借口

| 合理化借口                           | 现实                                                                   |
| ------------------------------------ | ---------------------------------------------------------------------- |
| "代码能跑，审查何必较真"             | 能跑的代码如果不可读、不安全、架构错了，就是在制造复利债务。           |
| "我自己写的，我知道没问题"           | 作者对自己的假设有盲区。每段代码都需要另一双眼睛。                     |
| "AI 生成的代码应该没问题"            | AI 代码需要更严格的审查——自信但可能出错，合理但不一定正确。            |
| "测试通过了，就是好的"               | 测试是必要但不充分的。测试不检查架构问题、安全问题、可读性问题。       |
| "这个 PR 太大了，但就是一次逻辑变更" | 超过 300 行的即使是单次变更也要求拆分。大 PR 隐藏 bug。                |
| "没发现问题，就是通过了"             | 明确写"未发现阻塞问题"。列出未覆盖的验证范围。不把"没测"说成"没问题"。 |

## 红线——停下来，按流程走

如果你发现自己在想：

- "这段改得挺对的，不用细看了"
- "没有发现问题"（但没有逐条过五轴检查）
- "这个 finding 太小了，不值得写"
- "审查太费时间，简单过一下就行"
- "我和作者关系好，不用太严格"

**以上这些都意味着：审查不完整。回到五轴框架，逐项评估。**

## 验证清单

审查完成后确认：

- [ ] 五轴（正确性/可读性/架构/安全/性能）均已评估
- [ ] 每条 finding 有文件路径和行号作为证据
- [ ] 严重度分级正确（CRITICAL/IMPORTANT/SUGGESTION/NIT）
- [ ] 变更规模已评估（超过 300 行已建议拆分）
- [ ] 新引入的依赖已审查
- [ ] 未覆盖的验证范围已明确列出
- [ ] 审查结果已按标准格式输出
