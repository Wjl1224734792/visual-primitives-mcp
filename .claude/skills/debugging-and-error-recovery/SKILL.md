# 系统化调试与错误恢复

## 概述

调试不是猜谜，是**假设驱动的调查过程**。本技能指导你用系统化方法定位根因，而非靠直觉改代码碰运气。

**核心原则：** 先理解问题，再尝试修复。修复若未对准根因，只会引入新 Bug。

**必需背景：** 你必须理解 `behavioral-guidelines`（准则 1：先思考再编码）。

## 何时使用

**适用场景：**

- Bug 复现与修复
- 线上故障排查
- 测试失败原因分析
- 性能劣化根因定位
- 异步/并发问题的排查
- 第三方服务调用异常
- 数据不一致问题

**不适用场景：**

- 已知原因的简单语法/类型错误（直接修复）
- 新功能开发（用 TDD，不用调试）
- 代码审查中发现的问题（记录 findings 即可，不在此阶段修复）

## 第一阶段：问题定义与复现

### 步骤 1：精确描述问题

在动手前先回答：

| 问题               | 示例                                     |
| ------------------ | ---------------------------------------- |
| **什么现象？**     | "用户登录后页面白屏，控制台报 500"       |
| **什么时间开始？** | "今天 14:30 部署后出现"                  |
| **影响范围？**     | "所有用户 / 仅新用户 / 仅 Chrome 浏览器" |
| **频率？**         | "每次必现 / 间歇性 / 偶发"               |
| **有无最近变更？** | "最近一次部署改了认证中间件"             |

### 步骤 2：创建最小复现

- 找到能稳定触发问题的最少步骤
- 不要依赖"大概率复现"——追求"100% 复现"
- 把复现步骤写成测试（如果可行），这是最可靠的复现方式
- 如果无法稳定复现 → 增加日志，等待下次出现

### 步骤 3：确认复现环境

- 问题在哪些环境出现？（dev / staging / production）
- 问题在哪些环境不出现？
- 环境差异是什么？（配置、数据量、依赖版本、网络拓扑）

## 第二阶段：根因追踪

### 方法 1：二分定位法

当改动的范围很大时：

```
1. 找到"最后一次正常"和"第一次异常"的提交
2. 取中间点，测试是否异常
3. 如果异常 → 问题在前半段，继续二分
4. 如果正常 → 问题在后半段，继续二分
5. 重复直到定位到具体提交
```

### 方法 2：差异分析法

当问题在特定环境出现时：

- 对比正常环境和异常环境的差异（配置、数据、版本）
- 逐一排除差异变量
- 不要同时改多个变量——改了也不知道哪个是原因

### 方法 3：调用链追踪

当问题在复杂系统中时：

- 从错误发生点向上追溯调用链
- 检查每层的数据变换（输入是否被意外修改？）
- 检查每层的错误处理（是否有异常被吞掉？）
- 使用日志、断点、或分布式追踪工具

### 方法 4：日志探针法

当无法用调试器时：

- 在疑似问题路径上增加临时日志
- 日志包含：时间戳、函数名、关键变量值、调用来源
- 部署到问题环境收集数据
- 问题解决后**必须清理临时日志**

## 第三阶段：纵深防御

修复 Bug 后，必须构建三层防线防止回归：

### 第一层：测试防线（必须）

- 把复现步骤写成自动化测试
- 确认修复前测试红，修复后测试绿
- 这确保同一个 Bug 不会再次出现

### 第二层：运行时防护（推荐）

- 输入验证：在入口处拒绝非法输入
- 超时控制：所有外部调用必须有超时
- 重试策略：幂等操作可重试，非幂等操作不可重试
- 断路器：连续失败时快速失败而非雪崩
- 优雅降级：依赖服务不可用时返回缓存/默认值

### 第三层：监控告警（考虑）

- 关键路径增加错误计数器和延迟直方图
- 异常频率超过阈值时告警
- 告警包含足够上下文（trace id、用户 id、操作时间）

## 第四阶段：错误分类与恢复策略

不是所有错误都该被修复。先分类：

| 错误类型     | 示例                 | 策略                    |
| ------------ | -------------------- | ----------------------- |
| **瞬时错误** | 网络超时、临时过载   | 重试（指数退避）+ 超时  |
| **持久错误** | 无效输入、权限不足   | 返回明确错误码，不重试  |
| **资源错误** | 磁盘满、内存不足     | 告警 + 降级 + 通知运维  |
| **逻辑错误** | 数据不一致、状态机错 | 修复代码 + 数据修复脚本 |
| **并发错误** | 竞态条件、死锁       | 修复同步逻辑 + 并发测试 |

### 错误恢复原则

1. **快速失败 > 错误传播**：发现不可恢复错误时立即返回，不要让脏数据扩散
2. **明确错误类型**：不要用通用 Error——区分 ValidationError、AuthError、TimeoutError
3. **不吞异常**：catch 后要么处理，要么重新抛出。空的 catch 块 = 埋雷
4. **降级有度**：降级返回的数据必须是安全的（不会导致下游做错误决策）
5. **重试幂等**：只有幂等操作可以安全重试。非幂等操作重试前检查是否已执行

## 第五阶段：基于条件的等待

调试异步/并发问题时，不要用 `sleep(N)` 猜测等待时间：

### 正确做法：条件等待

```typescript
// ❌ 错误：猜测等待时间
await sleep(1000);
expect(result).toBe(expected);

// ✅ 正确：等待条件满足
await waitFor(() => expect(result).toBe(expected), { timeout: 5000 });

// ✅ 正确：轮询检查条件
while (Date.now() - start < timeout) {
  const status = await checkStatus();
  if (status === 'completed') break;
  await sleep(100); // 短间隔轮询
}
```

### 条件等待规则

- 使用明确的超时时间（默认 5-10 秒）
- 轮询间隔尽量短（100-200ms），尽快发现条件满足
- 超时后给出清晰的失败信息（"等待 X 秒后仍未满足条件 Y"）
- 生产代码使用事件驱动而非轮询

## 常见借口

| 合理化借口                                    | 现实                                                             |
| --------------------------------------------- | ---------------------------------------------------------------- |
| "加个 sleep(2000) 就能过了"                   | sleep 掩盖问题而非解决问题。不同的机器、不同的负载、结果不一样。 |
| "试了几种改法，这个凑效了，就这样吧"          | "凑效" ≠ "修好"。你不知道为什么凑效，下次改代码还会坏。          |
| "这个问题偶发，不好复现，先加 try-catch 兜底" | try-catch 掩盖错误不是修复。先加日志定位根因。                   |
| "我把日志关了，没人会看"                      | 关键路径日志是生产环境的黑匣子。关了 = 下次出问题抓瞎。          |
| "修完了，测试通过，不用加回归测试"            | 不加回归测试 = 明知道这个 bug 可以再次出现但不设防。             |
| "这只是个边界条件，概率很低"                  | 低概率 × 大规模 = 高频事故。边界条件是 bug 高发区。              |

## 红线——停下来，按流程走

如果你发现自己在想：

- "改几行试试，不行再改回来"
- "这个 bug 看起来和上次那个差不多，用同样的方法修"
- "问题可能是环境问题，不是代码问题"
- "加个 try-catch 包住整个函数就行"
- "问题不复现了，应该是修好了"（但没有加回归测试）
- "这个 setTimeout 设长一点应该就不报错了"

**以上这些都意味着：你没有在做根因追踪。回到第二阶段。**

## 验证清单

修复完成后确认：

- [ ] 问题已通过自动化测试锁定（复现 → 修复 → 回归）
- [ ] 根因已定位（不是"改了 X 后不报错了"，而是"因为 Y 导致 Z"）
- [ ] 修复方案是最小的（未顺便重构或改无关代码）
- [ ] 相关调用链已检查（修复未引入新的边界问题）
- [ ] 三层防线已构建（测试 + 运行时防护 + 监控）
- [ ] 同类问题已排查（`grep` 搜索代码库中是否存在相同模式）
- [ ] 若是线上问题：已记录故障时间线、根因、影响范围、修复措施
