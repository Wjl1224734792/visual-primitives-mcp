# 规划与任务分解

## 概述

好的任务分解让实现阶段并行高效、风险可控。坏的任务分解让实现阶段互相阻塞、频繁返工。本技能提供将需求文档转化为可执行任务包的方法论。

**核心原则：** 任务的最小单位是**可独立验证的用户价值**。一个任务 = 一个端到端的功能切片，而不是一层技术实现。

**必需背景：** 你必须理解 `spec-driven-development`（需求是任务的事实源）。

## 何时使用

**适用场景：**

- Gate A 通过后，需求文档已确认
- 开始将需求分解为可执行任务
- 评估任务的优先级、风险和依赖
- 制定多轮次的实现计划

**不适用场景：**

- 需求文档未通过 Gate A（不可跳过）
- 用户只要求探索而非实现
- 单文件微小的单步骤改动

## 任务分解流程

### 第一步：盘点需求条目

从需求文档中提取所有 `REQ-XXX` 条目，按优先级排列：

```
P0（必须现在做）：REQ-001 用户注册, REQ-002 用户登录
P1（应该现在做）：REQ-003 密码重置, REQ-004 邮箱验证
P2（可以以后做）：REQ-005 社交登录, REQ-006 两步验证
P3（暂时不做）：REQ-007 生物识别登录
```

### 第二步：确定切片策略

选择切片策略（三选一或组合）：

#### 策略 A：垂直切片（优先推荐）

每个任务交付完整的端到端功能路径。

```
✅ TASK-001: 用户注册（schema + API + UI）
✅ TASK-002: 用户登录（auth + API + UI）
✅ TASK-003: 密码重置（schema + API + UI）
```

#### 策略 B：契约优先切片

先定义 API 契约，前后端根据契约并行开发。

```
✅ TASK-001: 定义用户模块 API 契约
✅ TASK-002: 用户注册后端（基于契约）
✅ TASK-003: 用户注册前端（基于契约，可与 TASK-002 并行）
```

#### 策略 C：风险优先切片

最不确定的部分先做，快速验证可行性。

```
✅ TASK-001: 性能关键路径 POC
✅ TASK-002: 第三方集成验证
✅ TASK-003: 后续常规功能
```

#### ❌ 反例：水平切片（禁止）

按技术层级拆分的任务。这导致：

- 前端要等后端、后端要等数据库设计
- 每个任务不能独立验证
- 集成时才暴露契约不匹配问题

```
❌ TASK-001: 设计全部数据库表
❌ TASK-002: 实现全部 API 端点
❌ TASK-003: 构建全部 UI 组件
```

### 第三步：分类任务类型

| 类型                | 说明                          | 实现策略                     |
| ------------------- | ----------------------------- | ---------------------------- |
| **DDD（领域驱动）** | 复杂业务逻辑、状态机、工作流  | 先建模后实现，侧重业务正确性 |
| **TDD（测试驱动）** | 算法实现、数据处理、API 逻辑  | Red→Green→Refactor           |
| **直接开发**        | 配置、UI 样式、文档、迁移脚本 | 无需 TDD，但需要验证         |

### 第四步：评估任务属性

每个任务填写：

```
### TASK-XXX
- 任务名称：
- 关联需求：REQ-XXX
- 类型：DDD / TDD / 直接开发
- 优先级：P0 / P1 / P2
- 预估变更行数：<200 行（超过需标注为风险任务）
- test_strategy：tdd / test_after / manual_only
- 依赖：[依赖的 TASK-XXX 或 API 契约]
- 被依赖：[被哪些 TASK 依赖]
- 风险等级：低 / 中 / 高
- 风险描述：（如适用）
- 完成标准：<可独立验证的验收条件>
```

### 第五步：识别并行机会

标记哪些任务可以并行执行：

- 两个任务互不依赖对方的输出 → 可并行
- 两个任务不修改同一个共享区域 → 可并行
- 不同 TDD 任务的 Red 步骤 → 可并行
- 契约优先模式下，前后端 → 可并行

## 任务属性模板

```
### TASK-001
- task_name: 用户注册功能
- requirement_ids: [REQ-001]
- type: TDD
- priority: P0
- estimated_lines: ~150
- test_strategy: tdd
- dependencies: []
- parallel_group: [TASK-002]  # 与登录功能无共享依赖，可并行
- risk: 低
- acceptance_criteria:
  1. 新用户可通过邮箱+密码注册
  2. 重复邮箱注册返回 409
  3. 密码强度不足返回 422
  4. 注册成功后发送验证邮件
```

## 风险任务标注

以下情况必须标注为风险任务：

| 条件                                         | 风险等级 |
| -------------------------------------------- | -------- |
| 预估变更超过 200 行                          | 中       |
| 预估变更超过 500 行                          | 高       |
| 涉及共享区域变更（schema、路由、契约、配置） | 高       |
| 涉及第三方服务集成                           | 中       |
| 涉及数据迁移                                 | 高       |
| 涉及并发/异步逻辑                            | 中       |
| 涉及支付/认证/安全                           | 高       |
| 在陌生代码库中工作                           | 中       |

## 多轮次规划

如果总变更超过 1000 行，必须分多轮次：

```
第 1 轮：MVP 核心功能（~400 行）
   TASK-001 用户注册
   TASK-002 用户登录

第 2 轮：增强功能（~350 行）
   TASK-003 密码重置
   TASK-004 邮箱验证

第 3 轮：扩展功能（~300 行）
   TASK-005 社交登录
```

## 常见借口

| 合理化借口                                       | 现实                                                                           |
| ------------------------------------------------ | ------------------------------------------------------------------------------ |
| "功能很简单，不需要分解任务"                     | 不分解就无法并行。2 个串行任务比 2 个并行任务多花一倍时间。                    |
| "水平切片也可以并行啊（同时做路由+服务+数据层）" | 水平切片并行会导致集成时大爆发——各方接口不匹配、重复实现、数据不一致。         |
| "任务多了管理起来麻烦"                           | 管理成本 << 返工成本。小任务出问题能快速定位，大任务出问题排查半天。           |
| "预估变更行数不准，没必要估计"                   | 不准也要估。估计能暴露过度复杂的任务——如果一个任务预估 300+ 行，说明它该拆了。 |

## 红线——停下来，按流程走

如果你发现自己在想：

- "这个功能按水平切片比较快"
- "任务不用分那么细，大概指个方向就行"
- "风险不用评估，遇到了再说"
- "多轮次太慢，一轮全做完"
- "依赖关系太复杂，随便排个顺序就行"

**以上这些都意味着：任务分解不完整。回到第二步，重新切片。**

## 验证清单

任务分解完成后确认：

- [ ] 所有 REQ-XXX 都至少映射到 1 个 TASK
- [ ] 任务使用垂直切片（或契约优先/风险优先）策略
- [ ] 无水平切片（按技术层级拆分的任务）
- [ ] 每个任务有明确的优先级和 test_strategy
- [ ] 依赖关系已明确（不会出现循环依赖）
- [ ] 并行机会已识别
- [ ] 风险任务已标注
- [ ] 单轮次总变更不超过 1000 行
- [ ] 共享区域已指定唯一责任方
- [ ] 每个任务有可独立验证的完成标准
