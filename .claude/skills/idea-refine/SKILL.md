# 想法细化

## 概述

用户通常带着模糊的想法来找你："我想做一个任务管理系统"、"优化一下首页性能"、"加个权限控制"。在进入正式需求文档之前，需要先用结构化提问把模糊想法变成可执行的范围。

**核心原则：** 模糊的想法 = 隐式的分歧。在规格中明确，可以避免实现时的返工。

**必需背景：** 你必须理解 `spec-driven-development`（想法细化后进入的下一阶段）。

## 何时使用

**适用场景：**

- 用户描述只有一句话或几个词（"加个导出功能"）
- 需求存在多种合理的解释（"权限控制"可以指 RBAC、ACL、OAuth 等）
- 验收标准不明确（"性能好一点" — 好多少？怎么测？）
- 范围边界模糊（"顺便把登录也优化一下" — 是顺便还是独立需求？）
- 用户自己也不确定具体要什么

**不适用场景：**

- 用户已经提供了详细的规格文档或 PRD
- 需求非常简单且无歧义（"把按钮颜色改成蓝色"）
- 用户明确说"不需要讨论，直接做"（尊重用户，但提醒风险）

**定位：** 这是流程的阶段 0 —— 在 `spec-driven-development` 之前，在 Jarvis 的"需求澄清"阶段之内。

## 细化流程

### 第一步：收集团队上下文

在向用户提问之前，先尝试获取事实依据：

- 装载 `code-explore-expert` 了解现有代码结构中是否有相关模块
- 检查项目已有的文档（ADRs、之前的需求文档）中是否有相关决策
- 只有在对现状有基本了解后，才能提出好问题

### 第二步：结构化提问

使用以下问题框架向用户提问。不必全部问完——挑选与当前想法最相关的 3-5 个。

#### 范围类问题

| 问题                                       | 目的          |
| ------------------------------------------ | ------------- |
| 这个功能面向谁？（用户角色/权限级别）      | 确定受众      |
| 最核心的一个用户旅程是什么？               | 找到 MVP 边界 |
| 哪些是明确不需要做的？（更重要的边界问题） | 防止范围蔓延  |
| 有参考竞品或类似产品吗？                   | 对齐期望      |

#### 行为类问题

| 问题                                                 | 目的               |
| ---------------------------------------------------- | ------------------ |
| 什么场景下触发这个功能？                             | 理解入口和上下文   |
| 成功后用户看到什么？失败后呢？                       | 定义正常和异常路径 |
| 有性能或规模上的预期吗？（用户数、数据量、响应时间） | 设定非功能需求     |
| 和现有功能的关系是什么？（替代 / 增强 / 独立）       | 确定集成点         |

#### 优先级类问题

| 问题                                 | 目的           |
| ------------------------------------ | -------------- |
| 这个功能的紧迫度如何？有截止日期吗？ | 排优先级       |
| 是否可以先做一个简化版，后续迭代？   | 切分 MVP       |
| 如果只能做一个子功能，哪个最重要？   | 极端优先级判断 |

### 第三步：输出细化结果

将提问结果整理为结构化摘要，格式：

```
## 想法细化结果

### 原始想法
<用户的原始表述>

### 澄清后的理解
- 目标用户：
- 核心用户旅程：
- 成功标准：
- 明确不在范围内：

### 开放问题（若有）
- [ ] <需要用户进一步确认的问题>

### 建议的下一步
→ 进入 spec-driven-development，围绕以下核心需求编写 PRD：
1. <核心需求 1>
2. <核心需求 2>
```

### 第四步：移交

- 将细化结果放在对话中让用户确认
- 用户确认后，以此为输入进入 `spec-driven-development` 流程
- 需求文档中应引用想法细化的要点作为背景

## 常见借口

| 合理化借口                               | 现实                                                                         |
| ---------------------------------------- | ---------------------------------------------------------------------------- |
| "用户说的很清楚了，不用细化"             | "加个权限控制"至少有 5 种理解。确认一种解释能省 10 倍返工时间。              |
| "问了用户也不一定答得上来"               | 答不上来本身就说明了风险——实现时你必然会遇到这些决策点，到那时来回问更耗时。 |
| "我先做一版出来，用户看到就知道要不要了" | 原型有原型的作用，但原型不能替代需求澄清。先澄清再做 = 一次做对。            |
| "这些边界条件实现的时候自然会考虑到"     | 实现时"自然"的考虑往往是"按自己理解"，而非"用户想要的"。                     |
| "细化花时间，不如直接开始写"             | 5 分钟提问能省 5 小时返工。想法细化是收益率最高的投资。                      |

## 红线——停下来，按流程走

如果你发现自己在想：

- "用户说的差不多了，开始写需求文档吧"
- "这个问题用户应该也不知道，我自己决定了"
- "问了 2 个问题用户都没回，先做了再说"
- "范围不重要，做出来再看"

**以上这些都意味着：想法细化不完整。回到第一步。**

## 验证清单

细化完成后确认：

- [ ] 已尝试获取代码库事实依据（code-explore-expert 或相关文档）
- [ ] 核心用户旅程已明确（至少 1 条）
- [ ] 成功标准可衡量（不是"好用一点"而是"操作步骤从 5 步减少到 2 步"）
- [ ] 明确不在范围内的内容（比"范围内"更重要）
- [ ] 开放问题已列出（如有）
- [ ] 细化结果已请用户确认
- [ ] 下一步已明确（进入 spec-driven-development 或继续澄清）
