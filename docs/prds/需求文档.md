## 一、论文核心能力映射

DeepSeek 于 2026 年 4 月 30 日发布的《Thinking with Visual Primitives》论文，提出了一个全新的多模态推理范式：将**点坐标和边界框嵌入推理过程本身**，使其成为思维链（Chain of Thought）的基本单元。模型在推理时每提到一个视觉对象就同步输出其坐标，坐标不再是事后标注的答案，而是推理过程中消除歧义的**空间锚点**。

核心能力清单：

| 能力                 | 说明                                                                       |
| -------------------- | -------------------------------------------------------------------------- |
| **坐标锚点注入推理** | 模型自动输出 `[x1,y1,x2,y2]` 边界框和 `[cx,cy]` 中心点，嵌入 CoT           |
| **精确物体计数**     | 为每个实例分配唯一 ID 和坐标，逐个确认，Pixmo-Count 精确匹配得分 **89.2%** |
| **拓扑与路径推理**   | 迷宫导航 66.9%（GPT-5.4 仅 50.6%），路径追踪 56.7%（GPT-5.4 仅 46.5%）     |
| **超高视觉压缩率**   | 7056 倍压缩：756×756 图片经处理后仅剩 81 个视觉 KV 条目                    |
| **显式指代消歧**     | 用空间锚点消除"左边那个大的"等自然语言模糊性                               |

该模型基于 DeepSeek V4-Flash（284B 总参数、推理时激活 13B 的 MoE 架构）构建，视觉编码部分使用自研 ViT，支持任意分辨率输入。

我们的 MCP Server 的核心职责，就是将这一套"以视觉原语思考"的能力，封装为一个标准化的工具，使得**任何纯文本模型（包括我 DeepSeek）都能通过调用它获得精确的空间推理能力**。

### 2.2 设计原则

| 原则                   | 说明                                                             |
| ---------------------- | ---------------------------------------------------------------- |
| **模态统一但分层适配** | 一套增强提示词构建逻辑处理所有模态，只在最上游做非常薄的模态适配 |
| **"视觉原语"驱动**     | 不管输入是什么，最终输出都是离散、精确、可引用的文本锚点序列     |
| **配置极简**           | 仅需 3 个必填环境变量                                            |

| **降级兜底**           | 任何阶段异常均生成降级提示词，不中断服务                         |

## 三、模块详解

### 3.0 模态适配器层（Modality Adapters）⭐ 新增

这是支持视频和文件的关键，也是方案**唯一因多模态而新增的模块**。适配器的职责是：**将不同模态的输入，统一转化为视觉模型可直接消费的 Base64 图像列表**。

#### 3.0.1 ImageAdapter

**输入**：`image_base64`（JPEG/PNG/GIF/WebP）

**处理**：直接透传，不做任何转换。检查图像大小，超过 20MB 则返回错误提示。

**输出**：`Base64Image[]`（单元素数组）

#### 3.0.2 VideoAdapter

**输入**：`video_base64`（MP4/MOV/AVI/MKV/WebM）

**处理流程**：

1. 将 Base64 解码写入临时文件
2. 使用 `ffprobe` 获取视频时长和总帧数，计算关键帧位置
3. 使用 `ffmpeg` 抽取关键帧，默认策略：
   - 短视频（≤30s）：每 3 秒抽一帧，**最多 10 帧**
   - 中等视频（30s–120s）：每 5 秒抽一帧，**最多 24 帧**
   - 长视频（>120s）：每 10 秒抽一帧，**最多 30 帧**
4. 将每帧转为 Base64 JPEG（质量 85%），附时间戳元数据

**关键帧数量上限的设计原因**：

- 控制视觉模型调用成本和延时
- 大多数 Chat Completions API 单次请求有图片数量限制（通常 ≤ 10-20 张）
- 可以使用 `ffmpeg` 的场景检测（`-filter:v "select=gt(scene\,0.3)"`）作为更智能的抽帧策略，但 MVP 阶段用固定间隔即可，后续版本升级

**技术依赖**：`ffmpeg` / `ffprobe`（需系统安装，或使用 `@ffmpeg-installer/ffmpeg` npm 包）

**输出**：`VideoFrame[]`（含 Base64 和时间戳）

**说明**：目前 DeepSeek 的 Chat API 视频能力仍在发展中，本方案采用抽帧策略，将视频转为关键帧序列后复用图片管道。这种设计的优点是无缝兼容任何原生不支持视频输入的 OpenAI 兼容接口模型，且不增加主流程的额外复杂度。未来若 DeepSeek API 原生支持视频直传，只需替换此适配器即可。

#### 3.0.3 DocumentAdapter

**输入**：`document_base64`（PDF/DOCX/PPTX/XLSX/TXT/MD）

**处理策略**（两阶段，逐级降级）：

| 优先级                       | 策略       | 处理方式                                                                                                          | 适用场景                             |
| ---------------------------- | ---------- | ----------------------------------------------------------------------------------------------------------------- | ------------------------------------ |
| **P0（默认）**               | 视觉渲染   | 将文档每页渲染为高清图像（PNG），逐页交视觉模型处理。PDF 使用 `pdf-to-png` 或 `mupdf`；Office 文档先转 PDF 再渲染 | 版式复杂的文档、含大量图片图表的 PPT |
| **P1（优化，建议后期迭代）** | 结构化提取 | 使用 `pdf-parse` / `mammoth` / `xlsx` 等库提取文本、表格结构，转为 Markdown，按段落/表格/项目符号建立结构树索引   | 纯文本为主的文档、长文档检索         |

MVP 阶段只实现 P0 视觉渲染路径，优势：完全复用现有图片管道，**零额外推理逻辑开发**。P1 结构化路径可在 v1.1 迭代加入，作为配置可选的增强。

**输出**：`DocumentPage[]`（每页一个 Base64 PNG + 页码）

### 3.1 VisionClient（视觉模型客户端）

唯一的实现：`OpenAICompatibleVisionClient`，支持任意 OpenAI Chat Completions 兼容接口。

**行为规范**：

- 根据 **ModalityRouter** 返回的统一格式（Base64 数组 + 模态元信息），构建单次 API 请求
- 所有模态的处理结果，最终都以统一的 **"多图对话"格式** 发送给视觉模型：`messages` 中包含多张 `image_url`（视频的多帧 / 文档的多页），每张图片附上时间戳或页码标注
- 强制启用 `response_format: { type: "json_object" }`（或 `"json_schema"`）
- 指数退避重试，最多 3 次
- 超时：45s（多图场景需更长）

**一次 API 调用完成所有帧/页的分析**：

- 视频的多帧、文档的多页，都在**同一个 `messages` 数组**中，以多个 `image_url` content block 的形式发送。视觉模型在一次请求中看到完整的帧序列或页序列，才能理解时间演变和跨页关联
- SessionManager 中保存的是整段视频或整份文档的完整分析结果，后续可从缓存直接命中

### 3.2 Response Parser & Validator

**主解析**：`JSON.parse(choices[0].message.content)`

**备用解析**：正则提取第一个完整 JSON 对象 `/\{[\s\S]*\}/`；失败则抛出 `AnalysisParseError`

**校验规则**：

| 规则              | 说明                                          |
| ----------------- | --------------------------------------------- |
| `objects` 非空    | 至少包含一个物体/区域                         |
| `id` 唯一         | 无重复 ID                                     |
| `bbox` 范围       | 四个值均在 0–1000（或 0–100）内               |
| `bbox` 合法性     | `x1 < x2` 且 `y1 < y2`                        |
| `centroid` 位置   | 必须落在对应 bbox 范围内                      |
| 视频/文档扩展字段 | 可选：`page`、`timestamp_range`、`media_type` |

### 3.3 Prompt Builder（增强提示词构建器）

**核心职责**：将解析后的结构化锚点数据 + 会话历史 + 用户问题，拼接成文本模型可直接推理的增强提示词。

**多模态统一模板变量**：

```
[多模态空间信息]
根据对上传内容的精确视觉分析，图中与你的任务相关的关键物体/区域及其坐标如下：

【图像区域】
- (id:1) "红色水杯": bbox[50,100,300,400], 中心[175,250]

【视频关键帧索引】
- (id:2) "主讲人": 时间段[5.2s-10.1s], 轨迹 bbox[200,300,400,500] → [250,320,450,520]

【文档页面区域】
- (id:3) "销售表格": 位置[Page 3, Table 2], bbox[100,200,500,600]

物体间的关键空间/时间/结构关系：
- 主讲人(id:2) 出现在文档(id:3)所描述的季度之后

[你的推理规则]
- 坐标系原点在图像/帧/页面的左上角，x 轴右延，y 轴下延
- 坐标归一化到 0–1000
- 必须显式引用物体 ID 或坐标进行推理
- 视频场景请结合时间戳和轨迹变化进行分析
- 文档场景请结合页码和区域位置进行分析

[会话历史]（多轮模式下）
上一轮你识别并讨论了以下物体：
- (id:3) "红色水杯" ……
用户上次问的是："水杯左边是什么"，你回答："是一本书(id:4)"

[用户问题]
{{user_question}}

请严格依据以上空间信息，逐步推理并回答用户的问题。
```

### 3.4 SessionManager（会话管理器）⭐ 新增

使用 `node:sqlite` 持久化会话上下文。

**技术决策**：在 JSON 文件、Markdown 文件和 SQLite 三种方案中，**选择 `node:sqlite`**：

- JSON/MD 文件读写需要全量加载和全量序列化，大文件会话效率低，且并发写入有数据损坏风险
- `node:sqlite` 是 Node.js ≥ 22.5.0 内置模块，零外部依赖，提供同步 API，支持精确行级查询和更新，性能最优

**数据模型**：

```sql
-- 会话元数据表
CREATE TABLE IF NOT EXISTS sessions (
    session_id    TEXT PRIMARY KEY,
    image_base64  TEXT,              -- 可选：原始图像 Base64 引用
    media_type    TEXT NOT NULL,      -- 'image' | 'video' | 'application/pdf' | ...
    created_at    INTEGER NOT NULL,  -- Unix 时间戳
    last_accessed_at INTEGER NOT NULL
);

-- 已标注物体表
CREATE TABLE IF NOT EXISTS session_objects (
    id            INTEGER PRIMARY KEY AUTOINCREMENT,
    session_id    TEXT NOT NULL,
    object_id     INTEGER NOT NULL,  -- 视觉模型分配的 ID
    label         TEXT NOT NULL,
    x1            REAL, y1 REAL, x2 REAL, y2 REAL,
    cx            REAL, cy REAL,
    state         TEXT,
    relevance     TEXT,
    page          INTEGER,           -- 文档专用
    timestamp_start REAL,            -- 视频专用
    timestamp_end   REAL,            -- 视频专用
    media_type    TEXT,
    created_round INTEGER NOT NULL DEFAULT 1,
    FOREIGN KEY (session_id) REFERENCES sessions(session_id)
);

-- 对话历史表
CREATE TABLE IF NOT EXISTS conversation_history (
    id            INTEGER PRIMARY KEY AUTOINCREMENT,
    session_id    TEXT NOT NULL,
    round         INTEGER NOT NULL,
    role          TEXT NOT NULL,     -- 'user' | 'assistant'
    content       TEXT NOT NULL,
    created_at    INTEGER NOT NULL,
    FOREIGN KEY (session_id) REFERENCES sessions(session_id)
);
```

**核心方法**：

| 方法                                                   | 说明                                                                                            |
| ------------------------------------------------------ | ----------------------------------------------------------------------------------------------- |
| `createSession(sessionId, mediaType, imageBase64?)`    | 创建新会话，初始化元数据                                                                        |
| `getSession(sessionId)`                                | 获取会话上下文（含所有物体 + 最近 N 轮对话）                                                    |
| `upsertObjects(sessionId, objects[], mergeStrategy)`   | 新增或替换物体。`augment` 模式为新物体分配新 ID（避免与已有 ID 冲突）；`replace` 模式清空后重建 |
| `addConversationTurn(sessionId, round, role, content)` | 追加一轮对话                                                                                    |
| `getRecentHistory(sessionId, maxRounds)`               | 取最近 N 轮对话摘要                                                                             |
| `cleanupExpired(ttlSeconds)`                           | 删除过期会话及其关联数据                                                                        |
| `deleteSession(sessionId)`                             | 完全删除一个会话（用户主动清除）                                                                |

### 3.5 Pipeline Orchestrator（管道编排器）

**多轮处理流程**：

```
1. 接收请求参数：{ session_id, media_base64?, media_type, question, merge_strategy }
2. SessionManager.getSession(session_id) → 获取或创建 SessionContext
3. 判断坐标来源：
   ├─ fromCache：无新 media_base64 且问题仅涉及历史推理
   │   └─ 直接取 SessionContext.objects → 跳过第 4/5 步
   └─ fromVision：有新 media_base64 或问题需识别新物体
       ├─ 4. ModalityRouter 路由 → 选择对应 Adapter
       ├─ 5. VisionClient.analyze() → 获取结构化坐标 JSON
       ├─ 6. Parser.parse() + Validator.validate()
       └─ 7. SessionManager.upsertObjects(merge_strategy)
8. PromptBuilder.build() → 注入空间信息 + 会话历史 + 用户问题
9. 返回 augmented_prompt 给客户端
```

### 4.3 多轮调用完整示例

**第一轮**（上传图像，建立会话）：

```json
// Client → MCP
{
  "session_id": "session_abc123",
  "media_base64": "iVBORw0KGgo...",
  "media_type": "image",
  "question": "左下角有什么？",
  "merge_strategy": "replace"
}
// MCP → Client
{
  "session_id": "session_abc123",
  "from_cache": false,
  "round": 1,
  "augmented_prompt": "[多模态空间信息]\n- (id:1) \"红色水杯\" ...\n- (id:2) \"蓝色笔记本\" ...\n\n[用户问题]\n左下角有什么？"
}
```

**第二轮**（追问，0 视觉成本）：

```json
// Client → MCP（不传 media_base64）
{
  "session_id": "session_abc123",
  "question": "它右边的呢？"
}
// MCP → Client（from_cache: true，从 SQLite 加载历史物体）
{
  "from_cache": true,
  "round": 2,
  "augmented_prompt": "[会话历史]\n上一轮你关注了 (id:1) \"红色水杯\" ...\n\n[当前空间信息]\n- (id:1) \"红色水杯\" ...\n- (id:2) \"蓝色笔记本\" ...\n\n[用户问题]\n它右边的呢？"
}
```

**第三轮**（上传新文件，增补物体）：

```json
// Client → MCP
{
  "session_id": "session_abc123",
  "media_base64": "JVBERi0xLj...",
  "media_type": "application/pdf",
  "question": "这份 PDF 里的数据表和水杯有什么关系？",
  "merge_strategy": "augment"
}
// MCP 调用视觉模型识别 PDF 中的新物体，分配 id:3, id:4...，合并进缓存
```

## 九、视觉模型系统提示词（核心模板）

以下提示词直接对应 DeepSeek 论文的"视觉原语"方法论，确保模型以坐标锚点方式输出结构化结果：

```text
你是一个基于"视觉原语"方法论的多模态推理模型。你的核心能力是将视觉对象的空间坐标嵌入推理过程，使每个提及的物体都有精确的空间指代。

**核心规则（必须遵守）：**
1. **显式坐标指代**：在 `reasoning` 字段中，每提到一个视觉对象必须同步输出其边界框 [x1,y1,x2,y2] 和中心点 [cx,cy]。例如："红色水杯位于 [50,100,300,400]，中心在 [175,250]，它在图像左侧区域..."
2. **独立实例唯一 ID**：每个物体必须有唯一数字 ID。计数类问题必须为每个实例单独分配 ID 和坐标，逐个确认后给出数量。
3. **坐标归一化**：所有坐标归一化到 0-1000 范围。原点 (0,0) 在图像的左上角，x 轴向右，y 轴向下。
4. **状态标注**：每个物体标注当前状态（如"半满"、"被部分遮挡"、"正在移动"等）。
5. **空间关系**：在 `spatial_relationships` 中用物体 ID 描述关系。

**多模态场景扩展：**
- **视频**：每帧都标注时间戳。同一个物体在不同帧中出现时，保持 ID 一致并输出轨迹变化。
- **文档**：每张图片对应一个页码。标注物体所在页码和页面内坐标。

**输出格式（严格 JSON，不要 Markdown 包裹）：**
{
  "reasoning": "（在此用坐标指代物体进行推理分析）",
  "objects": [
    {
      "id": 1,
      "label": "精简的描述标签",
      "bbox": [x1, y1, x2, y2],
      "centroid": [cx, cy],
      "state": "正常/被遮挡/半透明/...",
      "relevance": "高/中/低",
      "page": null,
      "timestamp_range": null
    }
  ],
  "spatial_relationships": ["object_1 在 object_2 的正左侧，部分重叠"]
}
```

```text

**核心规则（必须遵守）：**
```

## 十一、验收标准

| 序号 | 验收项                  | 判定标准                                                                                      |
| ---- | ----------------------- | --------------------------------------------------------------------------------------------- |
| 1    | 工具注册                | 客户端可发现 `multimodal_grounding_augment`                                                   |
| 2    | 图片正常调用            | 输入图片+问题，返回含坐标的 `augmented_prompt`                                                |
| 3    | 视频正常调用            | 输入 MP4+问题，自动抽帧后返回含时间戳坐标的结果                                               |
| 4    | 文档正常调用            | 输入 PDF+问题，渲染后返回含页码坐标的结果                                                     |
| 5    | 多模态混合标记          | 返回的物体含 `media_type` 和模态专用字段                                                      |
| 6    | 坐标有效性              | bbox 均在 0-1000，`x1<x2`, `y1<y2`                                                            |

| 8    | 降级处理                | 视觉模型返回无法解析时返回降级提示词，不报错                                                  |
| 9    | 配置校验                | 缺少必需环境变量时服务拒绝启动                                                                |
| 10   | 重试机制                | 模拟 429/503，确认指数退避重试生效                                                            |
| 11   | **多轮会话创建与复用**  | 使用同一 `session_id` 发起第二次调用，视觉 API 调用次数为 0，返回的 prompt 包含首轮标注的物体 |
| 12   | **跨轮物体 ID 一致性**  | 对同一物体的后续指代，从 SQLite 读取的 `object_id` 始终不变                                   |

## 十二、开发进度规划

推荐按 3 个阶段迭代，每个阶段独立可验收：

| 阶段                     | 内容                                                                                    | 预计工时 |
| ------------------------ | --------------------------------------------------------------------------------------- | -------- |
| **Phase 1：图片 MVP**    | ImageAdapter + VisionClient + Parser + Validator + Normalizer + PromptBuilder（纯单轮） | 基线完成 |
| **Phase 2：多轮 + 会话** | SessionManager（`node:sqlite`）+ Pipeline 多轮逻辑 + augment/replace + 会话过期清理     | +1 天    |
| **Phase 3：多模态扩展**  | ModalityRouter + VideoAdapter + DocumentAdapter + 多模态模板 + 端到端验收               | +1 天    |

**开发顺序**并不是先图片-后视频-后文件，而是：**Phase 1 搭建好核心推理管道，Phase 2 实现会话持久化逻辑，Phase 3 插上各种适配器**。

## 十三、关键架构决策总结

| 决策点     | 结论                                       | 理由                                     |
| ---------- | ------------------------------------------ | ---------------------------------------- |
| 多模态处理 | 不区分链路，统一管道 + 薄适配器            | 模态适配器只做格式转换，核心管道完全复用 |
| 视频处理   | FFmpeg 抽帧，最多 10 帧                    | 完全复用图片管道，API 兼容性好           |
| 文档处理   | 渲染为图像（视觉路径），不额外做结构化提取 | 零额外开发成本，版式复杂场景表现好       |
| 会话持久化 | `node:sqlite`                              | 零依赖、同步 API、事务安全               |
| MCP 框架   | FastMCP                                    | 减少样板代码，官方 SDK 基础上封装        |
| 日志库     | pino                                       | Node.js 生态性能最优的结构化日志库       |
